# 风光储一体化架构模式(Wind-Solar-Storage Integrated Architecture Pattern)

## 为什么需要架构模式

风光储一体化项目面临一个结构性问题：**三种能源类型的 SCADA/监控系统通常是独立建设的**（风用 A 平台、光用 B 平台、储用 C 平台）。一体化平台需要在上述异构系统之上构建统一的数据视图和可视化呈现。

架构模式的选择取决于客户现有系统的成熟度、数据开放程度和对实时性的要求。

## 三大架构模式

### 模式 A：叠加可视化层（Data Federation）

```
  ┌───────────────┐
  │  一体化可视化层  │  ← 本系统 (仅渲染展示)
  │  (三维引擎/门户) │
  └───────┬───────┘
           │
     ┌─────┴─────┐
     │ 数据网关    │  ← 适配层：协议转换、数据聚合
     └─────┬─────┘
           │
  ┌────┬───┴───┬────┐
  │ 风  │ 光   │ 储  │  ← 现有独立 SCADA/EMS
  └────┘      └────┘
```

| 维度         | 说明                                    |
| ---------- | ------------------------------------- |
| **适用条件**   | 已有成熟 Wind SCADA、PV 监控、EMS，客户**不要求替换** |
| **数据流**    | 只读：从各系统读取数据，不回写控制指令                   |
| **实时性**    | 取决于各子系统接口刷新频率，通常 1~5 秒                |
| **优势**     | 实施快（3~4 月）、风险低、不改动现有系统                |
| **劣势**     | 功能受限于接口数据范围，无法深层集成                    |
| **报价量级参考** | 参考陆上风电 165 天 / ~~~ 万（纯可视化）            |

### 模式 B：统一数据底座（Data Lakehouse）

```
  ┌───────────────────┐
  │  业务应用层（可视化+AI）│
  ├───────────────────┤
  │  统一数据底座        │  ← 时序数据库 + 数据治理平台
  │  (InfluxDB/TDengine) │
  ├───────────────────┤
  │  数据采集层          │  ← 统一采集网关
  ├───────┬───────┬────┤
  │ 风    │ 光    │ 储  │  ← 直接从设备层采集
  └───────┴───────┴────┘
```

| 维度         | 说明                        |
| ---------- | ------------------------- |
| **适用条件**   | 客户现有系统能力弱，或愿意替换/升级        |
| **数据流**    | 从设备层或系统层抓取原始数据，统一清洗、治理、存储 |
| **实时性**    | 高（< 1 秒），因数据路径最短          |
| **优势**     | 完全掌控数据质量，AI 诊断效果好         |
| **劣势**     | 实施周期长（6~9 月）、涉及现场设备改造风险   |
| **报价量级参考** | 约模式 A 的 1.5~2x            |

### 模式 C：混合模式（Hybrid）

```
  ┌───────────────────┐
  │  一体化可视化层      │
  ├───────────────────┤
  │  数据网关 + 轻量底座  │  ← 优先走接口，必要时下沉
  ├───────┬───────┬────┤
  │ SCADA │ 光伏  │ EMS │  ← 尽可能走现有系统接口
  │   +   │ 监+控  │   + │
  │ CMS   │ 气象   │ BMS │     关键数据直接采集
  └───────┴───────┴────┘
```

| 维度 | 说明 |
|------|------|
| **适用条件** | 部分系统能力强（如风电 SCADA），部分需下沉 |
| **数据流** | 混合：成熟的走接口，不成熟的走采集 |
| **实时性** | 混合（1~3 秒） |
| **优势** | 折中方案，实施周期可控、灵活应对不确定接口 |
| **劣势** | 架构不够统一，长期维护复杂度上升 |

## 模式选择决策树

```
客户数据源接口是否成熟？
├── 全部成熟（有 API/SDK 文档）
│   └── 模式 A：叠加可视化层 ← 推荐 JBFG 首选
│        理由：需求范围仅含可视化，无 AI 诊断要求
│
├── 部分成熟
│   └── 模式 C：混合模式
│        条件：关键数据（如振动）需下沉采集时
│
└── 全部不成熟或需替换
    └── 模式 B：统一数据底座
         风险：周期最长，需确认客户接受度
```

## 在 JBFG 项目中的推荐

| 因素 | 判断 | 依据 |
|------|------|------|
| 需求范围 | **可视化为主** | 未提及 AI 诊断、预测性维护 |
| 客户接口成熟度 | **未知**（Q5 待澄清） | 需求澄清问卷中已列为关键项 |
| 实施周期 | 期望短 | 参考陆上风电 165 天 |
| 推荐模式 | **模式 A**（最优）→ **模式 C**（备选） | 理由见上表 |

**方案书策略**：建议以模式 A 为基线报价，同时在方案中说明模式 C 的扩展路径（"若客户需集成振动/音纹等深层数据，可升级为统一数据底座方案"），给客户选择空间。
