# 方案书撰写规范与章节结构调研

> 调研目标：为「冀北风光储数智化生产支撑与数字孪生解决方案」提供章节结构依据。
> 方法：三维调研 → 基于 JBFG 需求反推直觉大纲 → 交叉对照 → 最终骨架。

---

## 一、空手直觉大纲（基于 JBFG 需求反向推导）

> 在外部调研前，仅基于 JBFG 已有资产（109 功能 + 34 非功能指标 + 18 风险 + 4 知识领域）推导的直觉大纲。

```
第一章  项目理解与需求分析
  1.1 项目背景 — 冀北风光储基地概况
  1.2 客户需求范围 — 基于 500009687 §5.2 的业务全景
  1.3 需求规格映射 — 109 项功能 × 11 个业务模块
  1.4 核心挑战识别 — 百万测点实时接入、500 并发、1s 端到端延迟

第二章  总体技术方案
  2.1 方案定位与设计理念 — 数字孪生驱动的一体化生产支撑
  2.2 总体架构 — 边-云协同、四层架构（采集/平台/业务/展示）
  2.3 关键技术选型 — 时序库、三维引擎、AI 框架、GIS 引擎
  2.4 与客户现有系统的集成关系

第三章  一体化数智场站平台
  → 对应 F1：系统基础组件 + 可视化展示 + 数据接入 + AI 数字人

第四章  功率-能量协同优化系统
  → 对应 F2：稳态协同 + 暂态策略 + GOOSE 快速通信

第五章  构网型透明场站系统
  → 对应 F3：机组/场站并网性能 + 发电性能评价 + 无功潮流分析

第六章  智能运维系统
  → 对应 F4：升压站/光伏/输电/储能智能运维 + 运维任务管理

第七章  数字运维管理平台
  → 对应 F5：实时监控 + AI 诊断 + 智慧运维门户 + 生产运营

  → 对应 34 项非功能指标：性能/可靠性/安全性/可扩展性

  9.1 分阶段交付策略（4 个月总工期）
  9.2 里程碑与交付物
  9.3 团队配置

第十章  风险管控
  → 对应 18 项风险矩阵：技术/业务/实施

第十一章  报价与服务
```

## 二、调研 A：电力行业方案撰写规范

### 2.1 关键标准与规范

电力行业智能化运维方案的撰写，没有一份专门的「方案书格式标准」，但受以下多层规范约束：

| 类型         | 代表性文件                           | 约束内容              |
| ---------- | ------------------------------- | ----------------- |
| **行业标准**   | DL/T 2844-2024 继电保护智能运维检修导则     | 智能运维术语定义、功能边界     |
| **国家标准**   | GB/T 19963.1-2024 风电场接入电力系统技术规定 | 风电并网技术指标          |
| **国家标准**   | GB/T 19964-2024 光伏发电站接入电力系统技术规定 | 光伏并网技术指标          |
| **企业标准**   | 国家电网 Q/GDW 系列                   | 系统架构、安全分区、通信协议    |
| **企业标准**   | 南方电网 OSC2 体系                    | 智能运维系统功能架构        |
| **安全法规**   | 网络安全法、数据安全法、等保 2.0              | 数据安全、安全分区要求       |
| **电力二次安防** | 电监会 5 号令                        | 安全 I/II/III 区隔离策略 |

### 2.2 电力行业方案书的隐含规范

基于招标文件编制的常见模式推断：

| 约束          | 说明                                            |
| ----------- | --------------------------------------------- |
| **安全分区**    | 方案须明确安全 I 区（控制区）、II 区（非控制生产区）、III 区（管理信息区）边界  |
| **通信协议合规**  | 南向须支持 IEC 104/IEC 61850/Modbus；北向须支持 DL/T 860 |
| **等保要求**    | JBFG 为等保二级，须说明安全防护措施                          |
| **国产化**     | 涉及关键基础设施，优先国产方案（JBFG 客户已明确）                   |
| **MTBF 量化** | 电力系统方案书通常需包含可靠性量化指标                           |

### 2.3 电力方案书特色

普通 IT 方案书 vs 电力行业方案书的差异：

| 维度     | 普通 IT 方案     | 电力行业方案                             |
| ------ | ------------ | ---------------------------------- |
| 安全架构   | 等保即可         | 需明确安全分区 + 横向隔离 + 纵向认证              |
| 通信协议   | RESTful/gRPC | IEC 61850/GOOSE/IEC 104/Modbus 协议栈 |
| 实时性    | 秒级           | 毫秒级（暂态控制 ≤100ms）                   |
| 可靠性论证  | SLA 百分比      | MTBF + 可用性双重指标                     |
| 关键指标来源 | 内部定义         | 国标/行标逐条引用                          |

## 三、调研 B：解决方案书通用章节结构与最佳实践

### 3.1 标准章节模板（11 章详细版）

整合多个来源（技术文档写作规范、咨询公司方法论、软件行业实践）：

| 章节             | 核心内容                          |
| -------------- | ----------------------------- |
| **1. 文档概述**    | 目的、范围、受众说明；术语表                |
| **2. 背景与现状分析** | 业务背景、技术现状、痛点识别                |
| **3. 需求定义**    | 功能需求、非功能需求、约束条件、优先级           |
| **4. 问题分析**    | 核心问题定义、决策条件、选型考量维度            |
| **5. 方案对比**    | 2-3 个可行方案，优劣分析，选型理由           |
| **6. 解决方案详述**  | 逻辑架构图/运行流程图/数据架构图/部署拓扑/接口/数据库 |
| **7. 风险与应对**   | 风险识别矩阵、应对策略与责任人               |
| **8. 实施计划**    | 分阶段规划、里程碑、交付物                 |
| **9. 工作量评估**   | 开发/联调/测试全流程估算                 |
| **10. 验收标准**   | 功能验收条件、性能指标验证                 |
| **11. 附录**     | 术语表、参考文献、评审记录                 |

### 3.2 两大叙事框架

**框架一：SCQA（麦肯锡金字塔原理）**

```
Situation（情境）→ Complication（冲突）→ Question（问题）→ Answer（回答）
```

适用于：客户对自身问题认知模糊，需要先建立共识。

**框架二：需求驱动型（技术标书常用）**

```
需求理解 → 方案设计 → 实施计划 → 风险管控 → 报价
```

适用于：客户已明确需求范围、有详细技术规范书（JBFG 属于此类）。

### 3.3 2025-2026 最佳实践

| 要点        | 说明                       |
| --------- | ------------------------ |
| **逻辑闭环**  | 需求→目标→架构→功能→实施→验收，全链条不脱节 |
| **图表优先**  | 架构图、流程图、时序图、ER 图优于纯文字    |
| **数据驱动**  | 技术参数基于实测或基准数据，选型理由充分     |
| **可读性分层** | 面向客户减少术语、附术语解释           |
| **风险矩阵化** | 可能性 × 影响，应对措施具体到人和时间     |
| **版本管理**  | 版本号 + 变更记录，重大变更同步知识库     |

### 3.4 常见误区

- 需求与设计脱节——需求未落地到具体功能模块
- 技术描述模糊——缺少具体参数和选型理由
- 图表与文字不一致——模块命名不统一
- 风险评估空泛——如「需完善」「待确认」
- 面向客户的方案术语过多

## 四、调研 C：竞品/友商方案架构对标

### 4.1 华为数字能源——行业标杆

**定位**：智慧新能源解决方案（FusionSolar 9.0）

**架构**：「端-边-云」三层，「集团-区域-场站」三级

```
集团层(云) — 智能驾驶舱/KPI/视频统一调阅/大数据/虚拟电厂
    ↕
区域层(边) — 鲲鹏集控/视频管理/区域远程集控/分布式诊断
    ↕
场站层(端) — 融合网络/智能终端/无人机+AI/SCADA对接
```

**核心差异化**：
- AI 大模型贯穿全链路（盘古气象预测、设备预测性维护、智能消缺）
- 数字孪生贯穿「规-建-维-营」全生命周期
- 量化指标醒目：运维效率↑50%、年发电量↑3%、施工误差↓40%

### 4.2 华为方案书结构（推断）

基于公开材料反推其方案书叙事结构：

```
1. 行业趋势与挑战 → 新能源运维的痛点
2. 解决方案概述 → 端边云 + AI + 数字孪生
3. 详细方案 → 按生命周期展开（规划设计/工程建设/运维消缺/运营增效）
4. 关键技术与差异化 → AI 预测/数字孪生/储能安全/构网技术
5. 案例与量化收益 → 全球案例 + ROI 数据
6. 实施与服务 → 交付模式 + 生态伙伴
```

### 4.3 对标启示

| 华为做法          | JBFG 可借鉴                      |
| ------------- | ----------------------------- |
| 数字孪生贯穿全生命周期叙事 | 方案书中以「数字孪生」为叙事主线              |
| 大量量化指标        | 充分引用 34 项非功能指标 + 业界 benchmark |
| 「端边云」简洁架构图    | 设计三层架构图，一目了然                  |
| ROI 量化说服      | 计算少人化、效率提升等可量化收益              |
| 生态伙伴联合交付      | 注明自有能力边界 + 合作方                |

## 五、交叉对照：直觉大纲 vs 调研结果

| 直觉大纲（JBFG需求驱动） | 调研结果（外部模板） | 差异分析 |
|-------------------------|-------------------|---------|
| 按功能域分章（F1-F5 各一章） | 按架构层次分章 | **JBFG 功能域较细**：5 个功能域独立成章会使方案书过于庞杂。建议合并为「业务子系统」一章，或用「业务方案」+「技术方案」两级结构 |
| 非功能单独成章 | 非功能分散在需求/方案/验收中 | **JBFG 非功能指标多（34 项）**：单独成章合理，对标华为的量化指标展示 |
| 缺少「方案对比与选型」章节 | 调研强调必须有多方案对比 | **JBFG 直觉大纲缺失**：应增加技术选型对比（如三维引擎、时序库、GIS 引擎的对比分析） |
| 风险放在最后 | 调研有放前面（After 需求）和最后两种流派 | JBFG 风险矩阵完整，放实施方案后合理 |
| 缺少「验收标准」章节 | 调研独立成章 | **应补充**：将 34 项指标转化为验收条件 |

### 关键发现

1. **JBFG 直觉大纲功能域过细**——5 个功能域各成独立章节会导致方案书 11+ 章，篇幅失控
2. **缺少方案对比维度**——直觉大纲直接给方案，但最佳实践要求 2-3 方案对比再推荐
3. **叙事主线不清晰**——直觉大纲是需求平铺，缺少像华为那样的「数字孪生贯穿生命周期」的叙事主线

## 六、推荐方案书章节结构（融合版）

基于直觉大纲 + 调研校准 + 竞品对标，建议以下章节结构：

```
第一章  项目概述
  1.1 项目背景 — 冀北风光储基地与数字化转型需求
  1.2 方案定位 — 数字孪生驱动的一体化生产支撑
  1.3 术语定义

第二章  需求理解与分析（SCQA）
  2.1 现状与挑战 — 多源异构、实时性、可靠性瓶颈
  2.2 需求全景 — 从 500009687 §5.2 到 109 项功能
  2.3 核心指标 — 34 项非功能要求逐项承接

第三章  总体技术方案
  3.1 设计理念 — 「数字孪生贯穿、AI 使能运维」
  3.2 总体架构 — 端边云协同、四层架构图
  3.3 关键技术选型与对比 — 三维引擎/时序库/GIS/AI 框架
  3.4 安全架构 — 安全分区 + 等保二级

第四章  业务子系统方案（合并直觉大纲第三至七章）
  4.1 一体化数智场站平台（F1 关键功能）
  4.2 功率-能量协同优化（F2 关键功能）
  4.3 构网型透明场站（F3 关键功能）
  4.4 智能运维系统（F4 关键功能）
  4.5 数字运维管理平台（F5 关键功能）

第五章  系统非功能特性
  5.1 性能指标承诺 — 34 项指标逐项承诺值
  5.2 可靠性设计 — MTBF ≥10000h + 冗余策略
  5.3 安全性 — 等保二级 + 安全分区 + 数据安全
  5.4 可扩展性 — 多场站扩展能力

第六章  项目实施计划
  6.1 分阶段交付（4 个月）
  6.2 里程碑与交付物

第七章  风险管控
  → 18 项风险应对策略（技术/业务/实施）

第八章  报价与服务
  8.1 项目报价
  8.2 服务保障与 SLA

附录  术语表 / 参考标准 / 接口规范清单
```

## 七、与原始直觉大纲的差异总结

| #   | 差异                         | 原因                    |
| --- | -------------------------- | --------------------- |
| 1   | 功能域从 5 个独立章合并为「业务子系统方案」1 章 | 控制篇幅、避免冗余、便于面向客户阅读    |
| 2   | 新增「方案对比与选型」节               | 最佳实践强制要求              |
| 3   | 新增「验收标准」（融入非功能章）           | 竞品对标发现：华为等方案必含量化验收    |
| 4   | 将风险章节从倒数第二章前移              | 紧跟实施计划更符合逻辑递进         |
| 5   | 功能域精简化——每节只写关键功能           | 109 项全列不可行也不必要；精华提炼即可 |
