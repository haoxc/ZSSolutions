# 数智化

## 核心定义

> **数智化**不是"数字化 + 智能化"的简单叠加，而是一种**以数据为原料、以模型为引擎、以决策为产出**的新型运营方法论——它将业务问题转化为数据问题，再将数据问题的答案转化为业务行动。

三个否定，划出边界：
- **不是** IT 系统建设（信息化）—— 不关注"流程有没有上线"
- **不是** 数据可视化展示（数字化）—— 不满足于"人看见了"
- **不是** 单纯引入 AI 算法（智能化）—— 不追求"模型精度最高"，追求"决策质量最高"

数智化的**最终检验标准**只有一个：
> **系统产出的决策建议，是否被一线人员信任并执行？**

## 术语来源与演变

```
2015        2018        2022        2024
  │           │           │           │
  ▼           ▼           ▼           ▼
"互联网+"──▶"数字化转型"──▶"数智化"──▶官方定调
            (IT 视角)    (DT+AI 视角)   (国家电网白皮书)

关键节点：
- 2018-2020: 阿里等互联网公司开始用"数智化"替代"数字化"
- 2022:  能源行业逐步采纳，用以区分"有 AI"和"没 AI"的数字化
- 2024:  国家电网《数智化坚强电网建设白皮书》官方定调
- 2025:  工信部文件中"数智化"出现频率首次超过"数字化"
```

> **术语选择即战略表态**：写"数字化"意味着侧重基础设施和系统建设；写"数智化"意味着强调 AI 驱动和数据闭环。在方案书中使用"数智化"而非"数字化"，本身就在向客户传达能力定位。

## 内涵拆解：四层结构

```
          ┌──────────────────────────────────┐
          │          数智化 (Digital-Intelligence)         │
          ├──────────────────────────────────┤
  顶层    │  ① 数据采集与治理（底座）                    │
  术      │  └─ 采、存、通、治、用 五字诀                  │
          ├──────────────────────────────────┤
          │  ② 模型化（核心引擎）                         │
  中层    │  └─ 将业务问题映射为数学问题                    │
  道      │     诊断模型 / 预测模型 / 优化模型 / 决策模型       │
          ├──────────────────────────────────┤
          │  ③ 决策闭环（价值出口）                        │
  底层    │  └─ 模型输出 → 人确认 → 行动 → 反馈 → 模型迭代      │
  法      │     （没有闭环的数智化是"展示型数智化"）              │
          ├──────────────────────────────────┤
          │  ④ 组织适配（必要条件）                        │
  根      │  └─ 信任模型的文化、理解概率的管理、数据驱动的流程       │
          └──────────────────────────────────┘
```

四层缺一不可。**第 ④ 层是最大瓶颈**——多数项目在 ①② 层投入充分，到了 ③ 层人不敢信、不肯用，到 ④ 层组织根本不适应"机器给建议"的工作方式。

## 数智化的三种误用

| 误用 | 常见表述 | 纠正 |
|------|---------|------|
| **标签化** | "我们这个系统是数智化的" | 数智化是方法论，不是系统属性。一个系统可以服务数智化，但系统本身不是数智化 |
| **跳级化** | "直接上 AI，一步数智化" | 跳过第 ① 层数据治理，AI 模型就是"垃圾进垃圾出" |
| **展示化** | "看我们的三维驾驶舱，数智化" | 只做到第 ① 层（数据展示），缺失 ②③④ 层，本质仍是数字化 |
| **玄学化** | "数智化就是 AI 赋能万物" | 不是每个场景都需要 AI，能用规则解决的问题用规则更可靠 |

## 数智化 ≠ 智能化

| 维度 | 智能化 (AI-ization) | 数智化 (Digital-Intelligence) |
|------|-------------------|-----------------------------|
| **起点** | "有什么 AI 技术可用" | "业务有什么问题需要解决" |
| **终点** | 模型精度、指标 SOTA | 决策采纳率、业务效果提升 |
| **成功标准** | 模型准确率 95% | 运维人员信任模型建议并执行 |
| **失败形态** | 模型过拟合 | 模型准确但没人用 |
| **关键角色** | 数据科学家、算法工程师 | 领域专家、变革管理者、一线用户 |

> 数智化 = 智能化 + **组织变革**。没有组织变革的智能化只是实验室里的算法竞赛。

## 数智化的代价（做之前要知道的事）

| 代价项 | 说明 |
|-------|------|
| **数据治理成本** | 70% 的时间花在数据清洗、对齐、标注上，仅 30% 的时间真正做建模 |
| **模型维护成本** | 模型上线后需持续监控、重训、调优，不是"一次交付终身使用" |
| **信任建设成本** | 一线人员从"不信 AI"到"愿意试一试"到"习惯参考 AI 建议"，需要 6~18 个月 |
| **组织摩擦成本** | 当 AI 建议与专家经验冲突时，听谁的？需要新的决策流程和权责界定 |
| **机会成本** | 投入数智化意味着同期放弃其他方向的资源投入 |

## 在方案书中的使用建议

| 使用方式 | 适用场景 | 示例 |
|---------|---------|------|
| **作为愿景词** | 方案封面、战略定位章节 | "构建冀北风光储数智化生产支撑体系" |
| **作为能力界定词** | 技术方案章节 | "本方案聚焦数智化第①②层（数据底座 + 可视化），为第③④层预留扩展" |
| **避免过度使用** | 功能描述、交付清单 | 不要每个功能模块都贴"数智化"标签——会贬损术语的信誉 |

> **推荐 JBFG 方案书的术语策略**：封面用"数智化"定调愿景，正文用"数字化"描述本次交付范围，在扩展章节中说明"数智化能力预留"。
