# 可视化引擎选型对比(Visualization Engine Selection)

## 为什么需要专门讨论

新能源场站三维可视化不是标准的游戏/建筑可视化场景，有其特殊性：

| 场景特性 | 对引擎的要求 |
|---------|------------|
| 大规模场区（风场 40~150km²） | 大世界坐标精度、LOD 分层加载 |
| 大量重复对象（120 台风机结构一致） | 实例化渲染、批量更新能力 |
| 实时数据驱动（转速/偏航/变桨 ≤2s） | 数据绑定效率、状态切换流畅度 |
| 多源数据叠加（SCADA/视频/气象/GIS） | 多图层融合、坐标对齐 |
| 跨平台（大屏/PC/移动端） | 多平台发布、无插件访问 |

## 候选引擎概览

| 引擎 | 技术路线 | 渲染能力 | 数据对接 | 部署方式 | 学习曲线 |
|------|---------|---------|---------|---------|---------|
| **Unity** | 实时3D + C# | ★★★★ | ★★★★★ | 客户端/WebGL | 中等 |
| **Unreal Engine** | 实时3D + C++/BP | ★★★★★ | ★★★ | 客户端为主 | 陡峭 |
| **Three.js** | WebGL 原生 | ★★★ | ★★★★ | 纯 Web | 较缓 |
| **Cesium** | 地理空间WebGL | ★★★ 3D | ★★★★ | 纯 Web(地理) | 中等 |
| **自研 WebGL** | 定制渲染管线 | ★★~★★★★★ | ★★★ | 纯 Web | 最陡 |

## 核心对比维度

### 渲染质量

```
照片级 ◄──────────────────────────────► 示意级
  UE5 (Nanite/Lumen)
    Unity (HDRP)
      Cesium (地球级)
        Three.js
          自研引擎
```

- UE5 照片级渲染对能源行业场景**过度设计**——风机叶片不需要 Nanite 级精度
- Three.js 足以满足"工业级"而非"照片级"要求
- **关键判断**：新能源可视化追求的是**数据清晰度**而非**视觉真实感**，引擎渲染质量在达到"工业级"后边际效益递减

### 大规模场景性能

| 指标 | Unity | UE | Three.js | Cesium |
|------|-------|----|----------|--------|
| 12 万面场景帧率 | 60fps | 60fps | 55fps | 50fps |
| 100 万面场景帧率 | 45fps | 40fps | 30fps | 25fps |
| 120台风机实例化 | ✅ 原生支持 | ✅ | ⚠️ 需手动优化 | ⚠️ |
| 地形 LOD | 需插件 | ✅ 内置 | 需库 | ✅ 原生 |
| Web 端帧率衰减 | 约 40% | 约 60% | 约 10% | 约 10% |

> Unity 在 WebGL 模式下帧率衰减明显（约 40%），但可通过预编译 + 纹理压缩缓解。
> Three.js 纯 Web 路线无转换损耗，但仍受浏览器单线程限制。

### 数据对接能力

| 能力 | Unity | UE | Three.js | Cesium |
|------|-------|----|----------|--------|
| WebSocket 实时数据 | ✅ | ✅（需插件） | ✅ 原生 | ✅ 原生 |
| REST API | ✅ | ✅ | ✅ 原生 | ✅ 原生 |
| GIS 坐标（WGS84） | ⚠️ 需计算 | ⚠️ 需计算 | ✅ 库支持 | ✅ 原生 |
| 视频纹理（RTSP） | ✅ | ✅ | ⚠️ 需转换 | ⚠️ 需转换 |
| D3/图表叠加 | ⚠️ | ❌ | ✅ 原生 | ❌ |
| 大屏多视口 | ✅ | ⚠️ | ✅ 原生 | ⚠️ |

## 混合推荐方案（JBFG 首选）

考虑到 JBFG 项目需求以"可视化展示"为主、无插件访问是硬需求、工期 5~6 月的约束：

```
┌────────────────────────────────────────────┐
│           推荐架构：Three.js + Cesium 双引擎           │
├────────────────────────────────────────────┤
│                                            │
│  门户/大屏  ◄── Three.js ──► 在场站级展示             │
│  (轻量、快速、无插件)    风机实例化、数据面板、图表叠加       │
│                                            │
│  地理视口  ◄── Cesium ──► 在区域级展示                │
│  (GIS 坐标、地形、影像)  电站分布、周边环境、输电线路        │
│                                            │
└────────────────────────────────────────────┘
```

### 分工逻辑

| 场景 | 引擎 | 理由 |
|------|------|------|
| 区域概览（多电站地理分布） | Cesium | 原生 GIS 坐标、地形影像、无需坐标系转换 |
| 单场站三维漫游 | Three.js | 轻量、无插件、Web 原生、图表叠加方便 |
| 风机拆解与数据面板 | Three.js | 实例化渲染 + 数据绑定 + HUD 叠加 |
| 大屏驾驶舱 | Three.js | 多视口、图表叠加、无缝集成 |

### 否决 UE/Unity 的理由

| 理由 | 说明 |
|------|------|
| **WebGL 帧率衰减不可接受** | Unity WebGL 帧率下降约 40%，UE 约 60%，对数据实时性要求高的场景不可行 |
| **客户端部署不切实际** | 要求客户安装客户端在电网项目中权限审批复杂 |
| **渲染质量过度设计** | 新能源可视化需要的是"数据清晰"，不是"画面精美" |
| **团队维护成本高** | Unity/UE 需专职引擎开发，而 Three.js 前端团队可共用 |

### 特殊情况：何时选 Unity/UE

如果客户明确要求以下任意一项，回到 Unity 是合理选择：

- **照片级渲染**写入合同验收标准
- **VR 沉浸式巡检**作为核心卖点
- **离线高精度仿真**（非数据驱动，纯动画展示）
- **单机大屏专用**（无需 Web 分发）
