## 五、与 JBFG 项目的映射

| GIS 能力需求点       | 自研方案             | 采购超图/中地方案   | 结论             |
| --------------- | ---------------- | ----------- | -------------- |
| 地理坐标显示（WGS84投影） | Cesium 原生支持      | ✅           | 自研可覆盖          |
| 场区三维地形重建        | Three.js + DEM数据 | ✅           | 自研可覆盖          |
| 空间分析（缓冲区/叠置分析）  | 需自研              | ✅ 平台自带      | **采购更优**（复杂度高） |
| 坐标转换与配准         | 需自研              | ✅ 平台自带      | **采购更优**       |
| 可视化渲染           | Three.js 灵活可控    | ⚠️ 超图渲染能力有限 | **自研更优**       |

**建议**：核心可视化展示自研（Three.js + Cesium），空间分析功能可集成超图 iServer 做后端支撑（采购 ¥60万级服务器许可），形成"自研前端 + 采购后端"的混合路线。

## 六、GIS 平台能力架构调研

> 调研目的：系统梳理主流 GIS 平台的**能力分层架构**，为 JBFG 方案书技术架构设计（特别是 GIS 服务与平台能力的关系）提供架构级对标。
> 数据来源：各厂商官网技术白皮书、官方文档、公开技术博客。

### 一、总览：GIS 平台能力架构的四种模式

GIS 平台从技术架构视角可分为四种典型模式，对应不同的架构风格和适用场景：

```
GIS 平台能力架构四种模式
┌─────────────────────────────────────────────────┐
│  模式 A：传统分层架构                              │
│  (MapGIS 九州 / 早期 SuperMap)                     │
│  适用：单体部署、标准 GIS 服务场景                   │
├─────────────────────────────────────────────────┤
│  模式 B：云边端一体化架构                           │
│  (SuperMap 2024/2025 — BRT-IDC)                  │
│  适用：云原生部署、大规模分布式场景                   │
├─────────────────────────────────────────────────┤
│  模式 C：中台化架构                                │
│  (MapGIS 10.5 九州 — 技术中台+数据中台+业务中台)    │
│  适用：企业级平台、多业务复用场景                     │
├─────────────────────────────────────────────────┤
│  模式 D：领域网络架构                              │
│  (Esri ArcGIS Utility Network)                    │
│  适用：电力/燃气/水务等管网类场景，规则驱动拓扑建模      │
└─────────────────────────────────────────────────┘
```

**与 JBFG 的关系**：JBFG 平台的技术架构选择取决于客户规模和需求复杂度。初期单场站可参考模式 A 或 B；多场站统一管理时建议升级到模式 C 或 B 的云原生架构；如涉及配电网络精细化管理则模式 D 提供领域参考。

### 七、架构调研总结：对 JBFG 方案书的影响

#### 7.1 核心结论

1. **GIS 平台架构的发展趋势是从「工具」到「平台能力」**
   - SuperMap 的六大技术体系、MapGIS 的三大中台、Esri 的 Utility Network 都在证明：GIS 不再是独立的桌面工具，而是融入平台的能力层
   - JBFG 方案书中应将 GIS 定位为**平台能力组件**而非独立子系统

2. **「空间对象管理」是通用空白——USDF 是结构洞**
   - 四家厂商中没有任何一家将「设备对象注册与管理」抽象为独立平台层
   - SuperMap/MapGIS/EV-Globe 的「空间对象」停留在图层+要素模型
   - Esri UN 的 Feature Class 最接近对象概念，但受限于网络拓扑语义
   - **→ USDF 的品类定义空白被多方验证**

3. **三维渲染引擎的选择可独立于 GIS 平台**
   - EV-Globe Next 证明：通用数据模型（glb/gltf/3dtiles）使得渲染引擎与 GIS 服务可以解耦
   - SuperMap 的 S3M 标准和 MapGIS 的 M3D 标准虽是国产但互不兼容
   - **→ JBFG 的 Cesium + Three.js 双引擎选择正确，且应保持引擎无关层**

4. **AI 能力的「即插即用」定位是差异化优势**
   - SuperMap 的 AIF 和 MapGIS 的 IGServer-S 都提供了 AI 服务化能力，但都与自家平台紧耦合
   - JBFG/USDF 的 AI 即插即用（通过标准 API 供给数据）在开放性上更优
