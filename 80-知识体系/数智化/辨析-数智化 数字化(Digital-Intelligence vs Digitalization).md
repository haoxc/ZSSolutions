# 辨析-数智化 数字化(Digital-Intelligence vs Digitalization)

## 一句话分界

> **数字化**解决的是"从纸质到数据"——把业务流程搬上系统，让人看得见。
> **数智化**解决的是"从数据到决策"——让系统代替人看懂，并给出行动建议。

两者不是替代关系，而是**演进层级**：数字化是数智化的前提，数智化是数字化的跃迁。

## 三层演进模型

```
信息化 ──→ 数字化 ──→ 数智化
(IT化)    (DT化)    (AI化)

|            |            |
流程电子化    数据在线化    决策智能化
人找数据     数据找人     数据替人决策
```

| 层级 | 特征 | 场站运维场景示例 |
|------|------|----------------|
| **信息化** | 纸质单据→电子表单，Excel 管理 | 巡检记录从纸质本子变成电子表格 |
| **数字化** | 系统承载业务流程，数据全在线、可检索、可统计 | SCADA 数据实时采集，面板展示曲线，人工设阈值告警 |
| **数智化** | AI 模型替代人工分析，输出预测性决策建议 | 传动链振动数据→AI 诊断→"轴承剩余寿命 87天，建议第 80 天停机更换" |

## 四维对比

| 维度 | 数字化 | 数智化 |
|------|--------|--------|
| **核心目标** | 效率提升（降本） | 能力升级（增效 + 创值） |
| **数据处理** | 描述性分析（发生了什么） | 预测性+处方性分析（将发生什么、怎么办） |
| **技术栈** | 数据库、BI 报表、工作流引擎 | AI/ML、数字孪生、知识图谱、智能体 |
| **人机关系** | 人看系统 → 人决策 → 人执行 | 系统建议 → 人确认 → 系统/人执行 |
| **风险模式** | 系统故障 = 业务中断 | 模型偏差 = 错误决策（风险层级更高） |
| **度量标准** | 数据覆盖率、在线率、报表生成速度 | 预测准确率、误报率、自动处置占比 |

## 常见混淆

### "我们已经在做数智化"的错觉

| 误判 | 实际属于 | 识别特征 |
|------|---------|---------|
| 把 SCADA 升级为三维可视化 | **数字化**的展示升级 | 数据仍然是"让人看"，没有 AI 分析环节 |
| 上了 AI 告警规则引擎 | **数字化**的规则自动化 | 基于固定阈值/规则，非模型驱动 |
| 做了移动 APP 和 IPAD 驾驶舱 | **数字化**的渠道扩展 | 终端的移动化不改变数据流的本质 |
| ***真正的数智化信号***：系统中存在一个环节，**AI 模型产出结果后，人只需要确认或不确认，而非从原始数据中人工研判** |

## 在 JBFG 项目中的意义

客户需求范围目前落在**数字化**阶段——三维可视化、实时数据展示、巡检路线动态呈现，这些都是"让人看得更清楚"，而非"让 AI 看出人看不出的东西"。

方案书的策略选择：
- **策略 A (做实数字化)**：聚焦可视化展示层，不提或少提 AI 诊断，报价可控、交付风险低
- **策略 B (展示数智化能力)**：在方案中引入数智化能力图谱，虽非本次客户明确要求，但可体现差异化、为二期扩展埋线
- **策略 C (完整数智化方案)**：在报价中将 AI 诊断模块拆分为"本次可选"或"二期规划"，给客户选择空间

> 参考 [[数智化场站系统(Digital-Intelligence Station System)|数智化场站系统]] 末尾的"缺失的拼图"分析。

## 成熟度自评框架

一个团队/组织当前处于哪个阶段，可通过以下 5 个维度自评：

| 评估维度 | 信息化 (L1) | 数字化 (L2) | 数智化 (L3) |
|---------|-----------|-----------|-----------|
| **数据流** | 人工录入、Excel 传递 | 系统自动采集，统一存储 | 实时流式处理，数据闭环回馈 |
| **决策方式** | 全凭经验 | 看报表辅助决策 | AI 建议 + 人确认 |
| **告警机制** | 无/人工巡检 | 固定阈值告警 | 动态基线 + 趋势预测告警 |
| **运维模式** | 被动抢修 | 计划性维护 | 预测性维护（提前 N 天预警） |
| **人机协作** | 全人工 | 人操作机器 | 机器建议、人监管、机器执行 |

**判级标准**：只要有任一维度未达 L3，整体就不算数智化。数智化要求**所有维度同时到达 L3**。

## 投资与回报特征

```
回报
^
|       数智化 ◄── 高投入、长周期、非线性回报
|         ↗        （前期数据治理投入大，AI 模型见效需时间）
|       /  数字化 ◄── 中等投入、线性回报
|     /   （每加一个模块多一份效率提升）
|   / 信息化 ◄── 低投入、边际递减
| /
+──────────────────▶ 时间
```

| 维度 | 数字化 | 数智化 |
|------|--------|--------|
| **投资特征** | 按系统/模块线性投入 | 前期数据底座投入大，AI 建模持续投入 |
| **见效周期** | 3~6 月可见效率提升 | 12~18 月才见决策质量改善 |
| **ROI 可预测性** | 高（工时节省可量化） | 低（避免的损失难以精确计量） |
| **失败模式** | 系统上线却没人用 | 数据质量差 → AI 不准 → 失去信任 → 废弃 |
| **最大风险** | 需求不明确 → 系统做出来不匹配业务 | **数据治理半途而废 → 模型效果差 → 领导层失去耐心** |

## 组织能力要求

| 能力项 | 数字化团队 | 数智化团队 |
|--------|----------|----------|
| **核心角色** | 产品经理、前后端开发、UI/UX | 数据科学家、MLOps 工程师、领域专家 |
| **数据基础** | 业务数据化即可 | 需治理后的高质量标注数据 |
| **迭代模式** | 敏捷开发（2 周一迭代） | 数据+模型迭代（1~3 月一周期） |
| **管理层认知** | 看报表、在线率就够了 | 需理解概率预测、模型置信度、误报率 |
| **供应商关系** | 买系统、定制开发 | 买能力、联合建模、持续调优 |

**关键判断**：从数字化到数智化，团队构成需要**根本性重组**——多出数据科学家、MLOps 工程师、领域专家三个角色，少依赖纯前端/后端外包。

## 能源行业转型路径参考

### 典型路径（从数字化到数智化）

```
第 1 阶段：数据基础设施（3~6 月）
  └── 统一数据采集、清洗、时序存储，打通数据孤岛

第 2 阶段：可视化与规则告警（3~4 月）
  └── 三维数字孪生展示，固定阈值/规则告警 → 但此时仍在数字化阶段

第 3 阶段：AI 模型试点（4~6 月）
  └── 选 1~2 个高价值场景做 POC（如传动链诊断、叶片诊断）

第 4 阶段：模型规模化与闭环（6~12 月）
  └── 模型接入运维流程，工单自动派发，预测性维护常态化
```

### 常见陷阱

| 陷阱 | 表现 | 后果 |
|------|------|------|
| **跳级陷阱** | 数字化未完成就上 AI | 数据质量差 → 模型不准 → 失去信任 → 退回原状 |
| **展示陷阱** | 三维可视化很好看，但没有 AI 分析 | 仍处于数字化，容易误判为数智化 |
| **POC 陷阱** | 做了 10 个 AI POC 都没有工业化 | 模型在实验室准，在生产环境不准 |
| **人机失调** | 模型建议被运维人员无视 | 需同时做组织变革和信任建设 |

## 国际化映射

在跨语言方案书或与外资客户沟通时，"数智化"没有精准的英文对译。常见映射及局限：

| 英文术语 | 中译 | 与"数智化"的偏差 |
|---------|------|----------------|
| **Digitalization** | 数字化 | 只解决了"从纸质到数据"，不含 AI 决策环节 |
| **Digital Transformation** | 数字化转型 | 偏组织变革视角，技术层面止于数字化 |
| **AI-driven Transformation** | AI 驱动转型 | 接近但缺失了"数据底座"这一前提条件 |
| **Data-Driven Decision Making** | 数据驱动决策 | 只描述目标，未涵盖"模型化"这一中间环节 |
| **Intelligent Operations** | 智能化运营 | 偏运维领域，不具备通用方法论高度 |

**推荐方案书中的处理方式**：首次出现时写"数智化（Digital-Intelligence）"，后续统一使用"数智化"中文术语。不强行套用英文概念，避免读者用英语术语的既有认知来理解"数智化"——会窄化其内涵。

## 行业案例谱系

以下案例按"数字化→数智化"光谱排列，帮助判断不同项目在演进路线上的位置：

```
纯数字化 ◄──────────────────────────────────────► 完整数智化

SCADA 系统    三维可视化      规则告警      AI 诊断     决策闭环
  上线         平台         自动派单       + 建议      + 自动执行
  (L2)         (L2+)        (L2→L3)      (L3)        (L3+)
   │            │             │            │           │
   ▼            ▼             ▼            ▼           ▼
  华北油田     本项目的       广西电网     陆上风电     国网某省
  电站项目     JBFG 需求      智能告警     传动链诊断    无人变电站
              范围所在        项目         项目         (试点)
```

| 案例阶段 | 代表项目 | 在本项目中的参考价值 |
|---------|---------|-------------------|
| **纯数字化 (L2)** | 华北油田电站项目 | 最基础的 SCADA 可视化，可作为**及格线参考** |
| **展示升级 (L2+)** | **JBFG 本项目需求范围** | 三维可视化 + 数据面板，**当前定位在此** |
| **规则自动化 (L2→L3)** | 广西电网智能告警项目 | 虽用"智能"标签，但本质仍是阈值规则，不算真正数智化 |
| **AI 诊断试点 (L3)** | 陆上风电传动链诊断方案 | 真正进入数智化的门槛——有模型替代人工分析 |
| **完整数智化 (L3+)** | 国网某省无人变电站试点 | 含决策闭环 + 自动执行，当前能源行业前沿 |

> 根据[[../../02-参考资料/公司方案(及报价)|公司方案库]]中可查到的历史项目定位，**公司目前已完成项目全部集中在 L2-L2+（数字化阶段）**，L3 级数智化项目仅有方案（陆上风电）但尚无签约交付案例。JBFG 项目的方案书在设计数智化能力扩展路径时，需考虑公司自身的交付能力边界。

## 决策框架：项目该走哪条路？

### 快速判断表

在 JBFG 项目中，判断应主推数字化还是数智化，可以用以下 5 个问题快速定位：

| 问题 | 是 → 倾向数智化 | 否 → 倾向数字化 |
|------|---------------|---------------|
| 客户有治理过的历史数据（1 年以上）吗？ | 可考虑 AI 模型试点 | 先做数据底座，阶段 1 止于数字化 |
| 客户有明确的"人工做不好"的诊断场景吗？ | 数智化有价值锚点 | 数字化已能满足需求 |
| 客户管理层理解"模型输出是概率而非答案"吗？ | 组织层面有条件 | 需先做认知铺垫，否则数智化落不了地 |
| 项目预算充足（数智化通常多 50%~100%）？ | 可打包数智化模块 | 报价受限时优先保证数字化交付质量 |
| 项目周期允许 12 个月以上吗？ | 有足够数据积累时间 | 6 个月内交付的项目不适合做数智化 |

**JBFG 当前判断**：基于已知信息，第 1、3、5 项大概率答"否"，建议方案书以数字化为主线、数智化为能力预留。

### 方案书章节配置建议

| 方案书章节 | 如果客户偏数字化 | 如果客户偏数智化 |
|-----------|---------------|----------------|
| **项目定位** | "冀北风光储数字化生产支撑系统" | "冀北风光储数智化生产支撑系统" |
| **技术架构** | 模式 A（叠加可视化层） | 模式 B/C（统一数据底座预留） |
| **核心功能** | 三维可视化 + 数据展示 | 三维可视化 + AI 诊断模块 |
| **实施计划** | 165 天单期交付 | 分期：6月数字化 + 6月AI 试点 |
| **报价结构** | 平台开发费 + 集成费 | 平台费 + 数据治理费 + AI 建模费 |
| **交付团队** | 前端+后端+PM | 前端+后端+数据科学家+领域专家 |

## 方案书写作建议

如果在方案书中需要向客户阐述"为什么我们不一步到位做数智化"：

> **论据线：**
> 数智化的前提是高质量数字化。当前阶段先建设统一数据底座 + 三维可视化展示（数字化完备），为二期 AI 诊断模块铺设数据基础。跳过数字化直接做 AI 诊断，数据的完整性、一致性、时间对齐精度不足，模型效果无法保障。

这与 [[风光储一体化架构模式]] 中推荐**模式 A** 的逻辑一致——先做可视化层，预留数据底座扩展路径。

如客户明确表示对数智化能力感兴趣，可参考以下**分阶段路线图**进行沟通：

```
阶段 1 (0~6 月)：数字化底座建设 → 报价主体
  三维可视化 + 数据网关 + 实时监测面板
  → 客户感知：看得更清楚

阶段 2 (6~12 月)：数据治理 + 模型试点 → 增补合同
  统一数据底座 + 1~2 个 AI 诊断 POC
  → 客户感知：AI 开始替代人工判断

阶段 3 (12~18 月)：模型规模化 + 决策闭环 → 二期项目
  全模块 AI 诊断 + 运维工单自动派发 + 预测性维护
  → 客户感知：运维模式质变
```
