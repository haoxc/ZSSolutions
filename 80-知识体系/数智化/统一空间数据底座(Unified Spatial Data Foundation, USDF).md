# 统一空间数据底座 (USDF)

> 原称「一元化空间对象管理」。「底座」定位：它是地基，不是整栋楼——管空间参照系 + 对象注册 + 属性标准化 + 统一 API，不管业务逻辑。

---

## 一、USDF 核心定义

### 1.1 定位

```
业务系统层    ← 告警、工单、巡检、报表、AI 诊断（管业务逻辑）
    ↕ 统一 API
USDF 底座层   ← 对象注册、空间参照、属性标准化（管空间对象元数据）
    ↕ 数据管线
数据采集层    ← IEC104/Modbus/GOOSE/GB28181（管数据接入）
```

### 1.2 职责边界

| USDF 负责 | USDF 不负责 |
|----------|-----------|
| 所有设备/设施的全局唯一 ID 注册 | 告警规则引擎 |
| 统一坐标系统 + 层级关系（场站→区域→设备→部件） | 工单流转逻辑 |
| 每个对象的标准属性集（名称/类型/位置/状态/关联设备/三维模型引用） | AI 推理与模型训练 |
| 向上层业务系统提供统一查询 API | 巡检路线规划 |
| 对象生命周期管理（创建/更新/退役） | 报表生成逻辑 |
| 空间拓扑关系（包含/邻接/连接） | 业务审批流程 |

### 1.3 对象粒度

运维关注的最小单元，非物理最小元件：

| 层级 | 示例 | 粒度 |
|------|------|------|
| 场站 | 冀北风光储基地 | 1 个 |
| 区域 | 风电区、光伏区、储能区、升压站区、输电线路区 | ~10 个 |
| 设备 | 单台风机、单台逆变器、单台储能 PCS | ~500 个 |
| 部件 | 风机叶片、逆变器功率模块 | 按需，非全量 |

**原则**：设备级全覆盖，部件级按需建模。避免百万级对象拖垮底座。

## 二、数据分层与流向

### 2.1 全链路数据流

```
场站设备
  ├─ 高频采集(1kHz/6.4kHz) ──→ Kafka(分区≥32) ──→ 时序库(ClickHouse)
  ├─ 稳态采集(1s)         ──→ Kafka ──→ 时序库 + 业务库(Postgres)
  └─ 视频流(RTSP/GB28181)  ──→ 流媒体服务器 ──→ 视频存储(NAS)
                                      │
                              ┌───────┴────────┐
实时查询(≤1s)              ↓                  ↓ 历史查询(≤5s)
  Redis 缓存层              展示层（前端API）     物化视图 + 预聚合
```

### 2.2 各层数据契约

| 层 | 职责 | 数据形态 | SLA |
|----|------|---------|-----|
| 采集层 | 协议适配、数据归一化、时间戳对齐 | 原始报文 → 标准化 JSON | 采集间隔 = 数据源频率 |
| 消息总线 | 多源汇聚、分区路由、持久化 | Kafka Topic（按场站+数据类型分区） | 消息延迟 ≤50ms |
| 时序库 | 测点历史数据、聚合查询 | ClickHouse（按时间+场站分区） | 写入 100万条/s；查询 ≤5s |
| 业务库 | 设备主数据、告警记录、工单、档案 | Postgres + PostGIS | 事务一致性 |
| 缓存层 | 实时快照、热点数据 | Redis（设备最新状态） | 命中 ≤1ms |
| 展示层 | API 聚合、权限过滤、格式转换 | RESTful API Gateway | P95 ≤200ms |

### 2.3 不可越层规则

- 展示层不直连时序库（必须通过 API Gateway）
- 采集层不直接写业务库（必须通过 Kafka）
- USDF 底座不存时序数据（时序数据走时序库，底座只存对象元数据）

## 三、实时/非实时分流

### 3.1 三通道分流模型

```
优先级 1 (P1) — 暂态控制通道
  协议: GOOSE (IEC 61850)
  延迟: ≤100ms
  流量: 风/光/储/SVG 紧急控制指令
  路径: 场站设备 → 边缘节点(直连) → 执行

优先级 2 (P2) — 稳态监控通道
  协议: IEC 104 / Modbus TCP
  延迟: ≤1s
  流量: 百万测点实时数据 + 设备状态
  路径: 场站设备 → Kafka → 时序库 → 缓存 → API

优先级 3 (P3) — 历史分析通道
  协议: RESTful API
  延迟: ≤5s
  流量: 聚合查询、报表、AI 推理请求
  路径: 物化视图 / ClickHouse 直接查询
```

### 3.2 分流策略

| 通道 | 资源隔离 | 降级策略 |
|------|---------|---------|
| P1 暂态 | 独立网段(VLAN) + 独立 GOOSE 交换机 | P1 通道中断 → 场站设备本地下垂控制兜底 |
| P2 稳态 | Kafka 独立分区 + 独立 Consumer Group | P2 通道拥塞 → 降采样（1s→5s），保证核心测点 |
| P3 历史 | 读写分离、查询超时熔断 | P3 超时 → 返回预聚合近似值 + 提示缩小时间范围 |

## 四、边云协同策略

### 4.1 职责划分

```
┌─ 中心云 ─────────────────────────────┐
│  AI 模型训练 / 全局分析 / 跨场站对比    │
│  历史数据归档 / 集团级驾驶舱           │
│  场站间功率协同优化 / 虚拟电厂调度     │
└────────────┬─────────────────────────┘
             │ 广域网（4G/5G/专线）
┌────────────┴─────────────────────────┐
│  场站边缘                              │
│  暂态控制(GOOSE, ≤100ms)              │
│  本地实时缓存(Redis)                   │
│  本地告警(断网自主运行)                │
│  高频数据采集与预处理                  │
└──────────────────────────────────────┘
```

### 4.2 断网自持

| 指标 | 要求 |
|------|------|
| 边缘断网自持时间 | ≥2 小时（本地缓存全量实时数据） |
| 断网告警自主运行 | 场站边缘独立执行 P0 级告警（烟火/热失控） |
| 恢复后数据补齐 | 边缘缓存数据自动回传中心云，不丢数据 |

## 五、安全分区架构

### 5.1 电力二次安防分区

基于电监会 5 号令，等保二级基线：

```
安全 I 区（控制区）
  ├─ 暂态调频/调压控制(GOOSE)
  ├─ 构网型设备控制指令下发
  └─ 安全防误保护逻辑
        │
        ├── 正向隔离装置（物理单向）
        │
安全 II 区（非控制生产区）
  ├─ SCADA 数据采集
  ├─ 实时监控(360°)
  ├─ AGC/AVC/一次调频监视（只读）
  ├─ 透明场站评价计算
  └─ 智能运维 AI 推理
        │
        ├── 防火墙
        │
安全 III 区（管理信息区）
  ├─ 运维门户 / 值班管理 / 两票
  ├─ 报表 / 物资 / 档案
  ├─ 三维可视化 Web 服务
  └─ 大屏宣传展示
```

### 5.2 安全措施

| 措施 | 说明 |
|------|------|
| 正向隔离 | I → II 单向，物理隔离装置，仅允许数据上行 |
| 防火墙 | II ↔ III，≥6Gbps 吞吐，IPS/AV |
| 入侵检测 IDS | ≥3Gbps，国产可控 |
| 认证 | 统一 SSO + RBAC 多级权限 |
| 审计 | 全操作日志，不可篡改 |
| 数据安全 | 关键测点数据加密存储，传输 HTTPS |

## 六、多场站扩展模型

### 6.1 扩展策略

```
一期（冀北基地，1 个场站）
  单实例部署
  数据模型预留多场站字段（station_id）

二期（+N 个场站）
  逻辑多租户：一个平台实例，按 station_id 隔离
  USDF 底座：天然支持多场站层级（集团→区域→场站→设备）
  时序数据：ClickHouse 按 station_id + 时间分区
  业务数据：Postgres 按 station_id 水平分表或分区
```

### 6.2 扩展时不变项

- USDF 对象模型不变（场站作为顶层容器，设备和部件继承层级）
- API 契约不变（查询加 `station_id` 参数即可跨场站）
- 安全分区架构不变（每个场站各自有 I/II/III 区，中心云统一 III 区）

### 6.3 扩展时需增加的

- 跨场站聚合查询引擎（全局 KPI 对比、排名）
- 集团级驾驶舱（统一展示 N 个场站运行态势）
- 场站间协同调度（虚拟电厂、跨场站功率分配）

## 七、统一 API 与集成标准

### 7.1 南向接口

```
场站设备 ──→ 统一采集适配器 ──→ Kafka

适配器支持协议：
  IEC 104 (风机/光伏 SCADA)
  Modbus TCP (储能 EMS / 逆变器)
  GOOSE (暂态控制)
  RTSP / GB28181 (视频)
  MQTT (气象站 / 无人机)
  RESTful API (第三方系统)
```

### 7.2 北向接口

所有上层系统和外部系统通过统一 API Gateway 访问：

| 端点 | 用途 | 格式 |
|------|------|------|
| `/api/v1/objects/{id}` | 空间对象查询 | JSON, USDF 标准响应 |
| `/api/v1/objects/{id}/telemetry` | 测点实时数据 | JSON, 时间序列 |
| `/api/v1/objects/{id}/alarms` | 告警查询 | JSON, 分页 |
| `/api/v1/objects/{id}/3dmodel` | 三维模型引用 | GLB/3DTiles URL |

### 7.3 内部服务间通信

| 场景 | 协议 | 说明 |
|------|------|------|
| 微服务间同步调用 | gRPC | 低延迟，强类型 |
| 异步事件通知 | Kafka | 解耦，持久化 |
| 配置同步 | 配置中心（如 Nacos） | 热更新 |

## 八、技术栈国产化路径

| 层次 | 推荐方案 | 国产备选 | 优先级 |
|------|---------|---------|:---:|
| OS | Linux (CentOS/AlmaLinux) | 麒麟/统信 UOS | P1 |
| 数据库-时序 | ClickHouse | TDengine / DolphinDB | P0 |
| 数据库-业务 | Postgres + PostGIS | 达梦 DM8 + 空间扩展 / 人大金仓 | P0 |
| 缓存 | Redis | 华为 GaussDB(for Redis) | P1 |
| 消息队列 | Kafka | RocketMQ / Pulsar | P0 |
| API 网关 | Kong / APISIX | APISIX（已国产） | P1 |
| 三维引擎 | Cesium + Three.js（开源，已自主可控） | 无依赖 | ✅ |
| GIS 引擎 | GeoServer（开源） | 超图 iServer / 中地 MapGIS | P0 |
| AI 推理 | PyTorch / ONNX Runtime | 华为昇思 MindSpore / 百度 PaddlePaddle | P2 |
| 流媒体 | Nginx-RTMP / ZLMediaKit（开源） | 国产无强依赖 | ✅ |

**策略**：P0 项在方案书中明确推荐的国产方案；P1 项标注兼容性；P2 项标注为可选项。
