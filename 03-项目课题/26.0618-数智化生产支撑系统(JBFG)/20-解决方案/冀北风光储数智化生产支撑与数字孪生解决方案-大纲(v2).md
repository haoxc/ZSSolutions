> 方案定位：**USDF 统一空间数据底座——新能源场站的空间对象操作系统**。一台设备注册一次，所有业务按需消费。毫秒级实时驱动，AI 模型即插即用。

## 卷首语 · 致客户：一分钟读懂本方案

### 一句话方案

> **USDF 统一空间数据底座——新能源场站的空间对象操作系统。** 就像操作系统让应用程序不用关心底层硬件，USDF 让业务系统不用关心数据采集、设备协议和空间坐标。

### 四个「不一样」

| 不是… | 而是… |
|-------|-------|
| 三维可视化平台 | **空间对象操作系统**——对象注册一次，所有业务消费 |
| 功能堆砌 | **架构约束驱动**——可靠/扩展/AI 友好/实施效能四个硬约束决定全部技术选型 |
| 黑盒集成 | **开放 API 生态**——北向标准 API + 多语言 SDK，第三方可自主集成 |
| 一次交付 | **持续扩展的平台**——二期加场站不加字段、不改 API |

### 量化价值一览

| 价值维度 | 量化收益 | 推导依据 |
|----------|---------|---------|
| 运维人效 | 人工巡检频次降低 60%，运维人效提升 40%+ | AI 检出率 ≥80% + 告警准确率 ≥95% |
| 故障响应 | 暂态事件人工介入率降低 80% | 暂态控制 ≤100ms + 端到端 ≤1s |
| 发电损失 | 年故障损失电量降低 15–20% | 光伏故障识别 ≥97% + 储能四级预警 |
| 平台扩展 | 二期场站接入工作量降低 70% | 低代码组态 + `station_id` 预留 |
| 系统可用 | 年非计划停运 ≤0.88 次 | MTBF ≥10,000h + 断网自持 ≥2h |

## 第二章 平台建设思想与顶层设计

> ★ **本章是 v2 的核心新增章。** 回答「为什么这样做」——是方案书的思想锚点。先讲清楚思想，第三章再展开技术细节。客户读到这里就应该能向领导复述「他们方案的核心思想是什么」。

### 2.1 核心思想：空间对象操作系统

**定位：** USDF 统一空间数据底座——新能源场站的空间对象操作系统。一台设备注册一次，所有业务按需消费。

**OS 类比：**

| OS 概念 | USDF 对应 | 工程含义 |
|---------|----------|---------|
| 文件系统 | 空间对象注册 | 每个设备/设施获得全局唯一 ID + 空间坐标 |
| 设备驱动 | 统一采集适配器 | IEC104/Modbus/GOOSE → 统一数据格式，新协议插拔式接入 |
| 系统调用 | 北向 API | `/api/v1/objects/{id}/*` 标准接口，业务系统统一消费 |
| 进程调度 | 三通道分流 | P1 暂态 / P2 稳态 / P3 历史，按优先级调度计算和网络资源 |
| 用户态/内核态 | 业务层 / USDF 底座层 | 业务系统不碰数据管线，USDF 不管业务逻辑 |

**对象优先 vs 功能优先：**

```
传统 SCADA（功能树）              USDF（对象树）
  采集 ──→ 存储 ──→ 展示           场站 ──→ 区域 ──→ 设备 ──→ 部件
   ──→ 告警 ──→ 报表                 │        │        │        │
                                     ▼        ▼        ▼        ▼
                                  所有功能围绕对象组织，而非围绕管线组织
```

> 来源：[[平台建设思想-统一空间数据底座(USDF)]]# 一、设计动机

### 2.2 四大架构约束

四个属性不是并列的修饰词，而是**架构设计的硬约束条件**——每个约束都有检验标准：

```
可靠                    扩展                  AI友好               实施效能
  │                      │                     │                    │
GOOSE 暂态 ≤100ms     多场站天然支持          USDF 标准 API        边云协同
安全 I/II/III 区        station_id 预留        gRPC/Kafka           断网自持 ≥2h
三通道分流              逻辑多租户              数据就绪              国产化 P0 直选
```

| 约束 | 架构含义 | 检验标准 | 违反后果 |
|------|---------|---------|---------|
| **可靠** | 暂态控制独立通道，安全分区物理隔离 | GOOSE ≤100ms，I 区防误绝不妥协 | 暂态控制延迟 → 设备损坏 / 电网事故 |
| **扩展** | USDF 天然多场站，数据模型一期预留 `station_id` | 二期不加字段、不改 API | 二期推倒重来，一期投资沉没 |
| **AI 友好** | 北向 API 输出标准化对象 + 时序数据 | 算法团队不碰采集层 | AI 模型与数据管线紧耦合，无法独立迭代 |
| **实施效能** | 边云协同 + 断网自持 + 国产化 P0 直选 | 一期部署不依赖二期基础设施 | 基础设施未到位 → 项目延期 |

> 来源：[[平台建设思想-统一空间数据底座(USDF)]]# 二、方案定位

### 2.3 顶层原理一：三通道分流

**物理依据：** 电力系统的数据天然分三个物理层级，不可混为一谈。

| 通道 | 协议 | 延迟 | 用途 | 降级策略 | 物理约束来源 |
|------|------|------|------|---------|------------|
| P1 暂态控制 | GOOSE | ≤100ms | 风光储/SVG 紧急控制 | 中断 → 本地兜底 | 保护动作必须在 100ms 内完成 |
| P2 稳态监控 | IEC104/Modbus | ≤1s | 百万测点实时数据 | 拥塞 → 降采样 | SCADA polling 周期 1-5s |
| P3 历史分析 | RESTful | ≤5s | 聚合查询/报表/AI | 超时 → 近似值 | 报表和训练不需要实时 |

**为什么不能一视同仁？**
> 试图用同一套数据管道处理三种数据 → 要么暂态控制不可靠（安全风险），要么稳态监控延迟高（可用性差），要么两者兼顾成本爆炸。

> 来源：[[平台建设思想-统一空间数据底座(USDF)]]# 4.2

### 2.4 顶层原理二：不可越层规则

| 规则 | 理由 | 违反后果 |
|------|------|---------|
| 展示层不直连时序库 | 时序库面向写入优化，不适合高并发点查 | 大屏卡顿、API 超时 |
| 采集层不直写业务库 | 业务库需要事务一致性，高频写入冲击 | 主数据污染、事务失败 |
| USDF 不存时序数据 | USDF 管对象元数据，时序数据归数据平台层 | 底座膨胀、查询性能退化 |

> 来源：[[平台建设思想-统一空间数据底座(USDF)]]# 4.1 末尾

### 2.5 顶层原理三：空间是统一维度

五种对象（风机/光伏/储能/升压站/线路）共享一个属性：**在空间中占据位置，与周围对象存在空间关系。**

**对象粒度：** 设备级全覆盖，部件级按需建模。

| 层级 | 示例 | 数量级 |
|------|------|:-----:|
| 场站 | 冀北风光储基地 | 1 |
| 区域 | 风电区、光伏区、储能区、升压站区 | ~10 |
| 设备 | 单台风机、单台逆变器、单台 PCS | ~500 |
| 部件 | 风机叶片、逆变器功率模块 | 按需 |

**工程价值举例：**
- 「离 3# 风机最近的储能舱是哪个？」→ 一次空间查询，无需定制 JOIN
- 「输电线路走廊内有多少块光伏板？」→ GIS 缓冲区分析，毫秒级

> 来源：[[平台建设思想-统一空间数据底座(USDF)]]# 3.3

### 2.6 方法论：从指标反推架构

**原则：每一个架构决策都能追溯到一个可检验的非功能指标。** 这是方案技术诚意的体现——不是先选技术再找理由，而是先有指标再决定架构。

| 非功能指标 | → 架构决策 | 对应原理 |
|-----------|-----------|---------|
| 端到端 ≤1s | → 必须三通道分流 | 原理 2 |
| GOOSE ≤100ms | → 必须安全 I 区独立 | 原理 2 |
| 二期不加字段 | → station_id 预留（对象优先） | 原理 1 |
| 百万测点写入 | → ClickHouse + Kafka ≥32 分区 | — |
| AI 检出率 ≥80% | → USDF 标准 API（算法不碰采集层） | 原理 1 |
| 低代码组态 | → 平台能力四个支柱 | Q11 |

> 来源：[[../03-需求分析/非功能指标定义]] + [[平台建设思想-统一空间数据底座(USDF)]]# 四

### 2.7 平台能力的四个支柱 (Q11)

四项横向平台能力，跨越所有业务子系统，是「AI 友好 + 实施效能」约束的直接产物：

| # | 能力 | 核心价值 | 对应 Q |
|---|------|---------|:----:|

## 第三章 总体技术架构

### 3.1 设计理念与架构原则（概要）
- USDF 底座 + 五业务域：底座管空间对象，业务管业务逻辑
- 底座先行 → 业务渐进 → 多场站可扩展
- 架构原则继承四大约束（可靠/扩展/AI友好/实施效能）

### 3.2 逻辑架构（四层）

```
展示层    大屏驾驶舱 / PC 门户 / 移动 APP / 远程巡检台
  ↕ 统一 API Gateway
业务层    F1 数智场站 / F2 协同优化 / F3 透明场站 / F4 智能运维 / F5 数字运维

平台层    USDF（对象注册/空间参照/属性标准化/拓扑关系）+ 数据引擎
  ↕ 南向采集
采集层    统一采集适配器 (IEC104/Modbus/GOOSE/RTSP-GB28181/MQTT)
```

### 3.3 部署拓扑（三级 + 安全分区）

```
  ┌─────────────────────────────────────────────────────────┐
  │ 安全 I 区（控制区）      │ 安全 II 区（非控生产区）        │ 安全 III 区（管理信息区）│
  │                         │                              │                        │
  │ GOOSE 暂态控制 ≤100ms   │ IEC104/Modbus 实时采集        │ 三维 Web 门户           │
  │ 防误保护                 │ Redis 实时缓存               │ 运维工单/报表/大屏        │
  │ 本地兜底                 │ GPU 边缘推理（CV 巡检）       │ AI 训练结果下发           │
  │                         │ AGC/AVC 控制                 │                        │
  │                         │ 断网自持 ≥2h                 │                        │
  └─────────────────────────────────────────────────────────┘
        ↕ 正向隔离装置              ↕ 防火墙

  风机 SCADA / 光伏监控 / 储能 EMS / SVG & 升压站 / 视频摄像头 / 无人机
```

> 来源：[[平台建设思想-统一空间数据底座(USDF)]]# 4.3, # 4.4

### 3.4 数据分层与流向

```
场站设备
  ├─ 高频采集(1kHz/6.4kHz) ──→ Kafka(分区≥32) ──→ 时序库(ClickHouse)
  ├─ 稳态采集(1s)         ──→ Kafka ──→ 时序库 + 业务库(Postgres)
  └─ 视频流(RTSP/GB28181)  ──→ 流媒体服务器 ──→ 视频存储(NAS)
                                      │
                              ┌───────┴────────┐
                              ↓                  ↓
                         Redis 缓存层        物化视图 + 预聚合
                        (实时查询 ≤1s)        (历史查询 ≤5s)
```

| 层 | 职责 | SLA |
|----|------|-----|
| 采集层 | 协议适配、数据归一化 | 采集间隔 = 数据源频率 |
| 消息总线 | 多源汇聚、分区路由 | 消息延迟 ≤50ms |
| 时序库 | 测点历史、聚合查询 | 写入 100万条/s；查询 ≤5s |
| 业务库 | 主数据、告警、工单 | 事务一致性 |
| 缓存层 | 实时快照 | 命中 ≤1ms |
| 展示层 | API 聚合、权限过滤 | P95 ≤200ms |

> 来源：[[平台建设思想-统一空间数据底座(USDF)]]# 4.1

### 3.5 关键技术选型与对比

| 决策域 | 候选方案 | 推荐 | 理由 |
|--------|---------|------|------|
| 三维引擎 | UE5 / Unity / Three.js+Cesium | 待定 | 需权衡照片级精度 vs 国产化 vs Web 化 |
| 时序数据库 | InfluxDB / TimescaleDB / ClickHouse | 待定 | 百万测点写入基准测试 |
| GIS 引擎 | GeoServer / 超图 / 中地 | 待定 | Q7 客户需 GIS 服务由我方提供 |
| AI 框架 | 开源 CV + 自训练 / 调用大模型 API | 待定 | P15 误检率 ≤30% 约束 |
| 流媒体 | RTSP/GB28181 直转 / 转码服务 | 待定 | P24 300 视频终端 + 1000 监测终端 |
| API 网关 | Kong / APISIX | APISIX | 国产化优先 |

### 3.6 AI 引擎架构

**三层架构：** 边缘推理（场站 II 区 GPU）+ 云端训练（中心云 GPU 集群）+ 人机协同闭环

| 层 | 部署位置 | 职责 | 关键指标 |
|----|---------|------|---------|
| 边缘推理 | 场站 II 区（GPU 服务器） | 烟火识别/通道隐患/设备异常实时检测 | 单次推理 <500ms |
| 云端训练 | 中心云 III 区（GPU 集群） | 模型微调/数据标注/版本管理/AB 测试 | 新模型下发 ≤30min |
| 人机协同 | III 区（运维门户） | 告警确认/误报标注/效果评估 → 反馈至训练集 | 检出率 ≥80%，误检率 30%→15% |

**数据就绪策略：** 算法团队不碰采集层——所有训练和推理数据通过 USDF 北向 API 获取。模型即插即用。
**模型范围：** 一期覆盖 CV 巡检类模型（F4），架构为 NLP/大模型预留接入能力。

> 来源：[[平台建设思想-统一空间数据底座(USDF)]]# 4.8

### 3.7 数据治理体系

**四层治理：** 主数据管理 → 元数据目录 → 数据血缘 → 数据质量

| 层 | 职责 | 关键规则 |
|----|------|---------|
| 主数据 | 设备/设施全局唯一 ID，标准属性集 | USDF 3.2 职责边界——单点注册，全局消费 |
| 元数据 | 测点目录、数据字典、API 字段映射 | 新增测点自动注册，避免「暗数据」 |
| 血缘 | 采集→Kafka→存储→API 全链路溯源 | 字段级血缘，回溯任意 API 字段的上游数据源 |
| 质量 | 完整性/时效性/唯一性/一致性校验 | 质量 SLA 按通道分级：P1 > P2 > P3 |

> 来源：[[平台建设思想-统一空间数据底座(USDF)]]# 4.9

**内部：** gRPC（同步）/ Kafka（异步）/ Nacos（配置）

**SDK：** Python / Matlab / C++ 多语言 SDK，第三方可自主集成

## 第四章 业务子系统方案

> 五业务域详述。每节聚焦「功能范围 + 关键指标 + 与 USDF 的接口关系」。

### 4.1 一体化数智场站平台 (F1)

**核心功能 (P0-P1)：**
- 全要素数据建模 (F1.1.1) — 灵活对象化建模，不写底层代码
- 多类型数据处理 (F1.1.2) — 风电/光伏/储能/仿真多源接入
- 低代码组态 (F1.1.4) — 普通技术人员可配页面
- 三维可视化 (F1.2.1-5) — 风机/光伏/储能/巡检/大屏，照片级精度
- 高频数据接入 (F1.3.1) — 1kHz/6.4kHz/12.8kHz 录波

**关键指标：** 500 并发 / ≤1s 端到端 / 百万测点实时
**USDF 接口：** 所有设备通过 USDF 北向 API 获取对象模型和空间参照

### 4.2 功率-能量协同优化 (F2)

**核心功能：**
- 稳态多源有功/无功协同 (F2.1.3-7) — 减少弃风弃光、平抑波动
- 暂态快速调频/调压 (F2.2.3-6) — GOOSE 协议 ≤100ms
- 安全防误 (F2.1.8) — 指令异常/状态异常防误保护

**关键指标：** 调频偏差 ±1% / 测频精度 ≤0.003Hz / 频率采样 ≤100ms
**USDF 接口：** 通过 P1 暂态通道获取设备状态，控制指令经安全 I 区下发

### 4.3 构网型透明场站 (F3)

**核心功能：**
- 机组并网性能评价 (F3.1.1-7) — 跟网/构网双模式，7 类指标
- 场站并网性能评价 (F3.2.1-4) — AGC/AVC/惯量/一次调频/暂态支撑
- 发电性能评价 (F3.3.1-3) — 风电/光伏/储能发电量统一管理

**关键指标：** 数据采集 ≤3s / 单机评价 ≤10s / 场站评价 ≤30s
**USDF 接口：** 指标数据通过 USDF 统一对象模型对外暴露

### 4.4 智能运维系统 (F4)

**核心功能：**
- 升压站自动化巡视 (F4.2) — 无人值守、温度异常识别
- 光伏智能运维 (F4.3) — 逆变器诊断/组串分析/灰损评估/无人机巡检
- 输电线路智能运维 (F4.4) — 通道隐患 AI 识别/电子围栏/智能审图
- 储能区安全运维 (F4.5) — 内短路/热失控四级预警/一致性分析
- 运维任务管理 (F4.6) — 例行/特殊/专项/自定义运维 + 告警闭环

**关键指标：** AI 检出率 ≥80% / 光伏故障识别 ≥97% / 告警准确率 ≥95%
**USDF 接口：** AI 推理结果通过 USDF 北向 API 回写对象告警状态

### 4.5 数字运维管理平台 (F5)

**核心功能：**
- 360° 实时监控 (F5.1) — 全局/单站/设备三级
- 智能 AI 诊断 (F5.2) — 多维度告警 + 派单闭环
- 智慧运维门户 (F5.3) — 个人门户/任务中心/两票管理/值班管理
- 生产运营 (F5.4) — 报表/物资/档案

## 第五章 系统非功能特性

### 5.1 性能指标承诺
- 34 项指标逐项承诺值 + 分层 SLA（来源：[[../03-需求分析/非功能指标定义]]）
- JBFG 量化价值承诺：运维人效 ↑40%+ / 暂态人工介入 ↓80% / 发电损失 ↓15-20% / 二期接入工作量 ↓70%

### 5.2 可靠性设计
- MTBF ≥10000h 的架构保障：N+1 冗余 / 故障自动切换 / 数据多副本
- 质保期内持续统计 MTBF，期末达标（来源：RB5）

### 5.3 安全性
- 等保二级：身份认证/访问控制/安全审计/数据加密/入侵检测
- 电力二次安防：安全 I/II/III 区隔离 + 正向隔离装置

### 5.4 可扩展性设计
- **多场站统一调度**：`station_id` 天然隔离，二期不加字段不改 API
- **功能模块可插拔**：新场站接入无需重构平台，新业务模块低代码配置上线
- **低代码建模**：F1.1.4 低代码组态——普通技术人员可配置页面/定义模型/编排工作流

### 5.5 开放 API 与开发者生态
- 北向 API + gRPC（同步）+ Kafka（异步）
- Python/Matlab/C++ 多语言 SDK
- `/api/v1/objects/{id}/*` 标准 RESTful 接口，第三方系统可自主集成
