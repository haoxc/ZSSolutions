# V3 功能设计深度审计（修正版）

> 前提修正：**南瑞运维平台提供算法库（业务层面算法），三维运营平台通过接口调用。** 这意味着下垂控制、无功优化、电池诊断等业务算法不属于我方设计范围——我方功能设计的核心在三维渲染、AI 视觉推理、空间对象管理、API 集成和交互流程。

---

## 一、结论

**V3 方案书的功能设计深度问题，根因不是「内容写少了」，而是「设计前提未宣告 + 双方分工未划定」——导致评审专家用「全栈自研」的预期来审视方案，自然觉得 F2/F3/F4 的算法设计是空白。**

修正后诊断：我方真正需要纵深设计的功能域中，**三维可视化交互（§4.1）和 AI 推理管线（§3.5）已有骨架但缺血肉**；而 **南瑞侧负责的算法域，方案书需要把「为什么不展开」说清楚，而非假装不存在**。

---

## 二、全局设计前提：当前缺失

### 2.1 问题

V3 方案书中，「南瑞提供算法库、我方调用」这一全局设计前提，仅以间接方式出现：

| 出现位置 | 表述 | 问题 |
|---------|------|------|
| §1.1 项目背景 | 「南瑞平台在三个维度存在能力空白：(1) 三维可视化……(2) AI 驱动智能巡检……(3) 构网型透明场站评价与功率-能量协同优化」 | 暗示我方填补这三个空白，但未说明我方是「建平台调算法」还是「连算法一起做」 |
| §1.2 互补关系表 | 南瑞侧「← 告警/工单数据读取」，我方侧「功率-能量协同优化」 | F2 归我方，但算法来自南瑞——这个关键矛盾未解释 |
| §3.6 数据接口互通 | 南瑞→我方：告警/工单/台账；我方→南瑞：AI 告警/巡检结果/评价报告 | 6 条数据流，但没有「算法调用」方向的数据流 |

**后果**：读者默认我方自研 F2/F3 的全部业务算法 → 阅读 §4.2/§4.3 时发现只有 1-2 段 → 判定功能设计浅薄。

### 2.2 缺失的声明

方案书应当在 §1.2（方案定位）或 §3.0（设计理念开头）增加以下全局设计前提：

> **算法获取方式**：本方案不重复建设南瑞平台已有的业务算法库。功率协同优化（下垂控制、有功/无功分配优化）、构网型评价（惯量/调频/电压穿越评分模型）、储能电池诊断（内短路/热失控/一致性算法）等业务算法，由南瑞平台算法库对外提供标准 RESTful/gRPC 接口，我方三维运营平台通过接口调用算法结果，聚焦于：
> - 三维可视化渲染与空间交互
> - AI 视觉推理管线（烟火识别/通道隐患/行为识别——这是我方的自有算法域）
> - 空间对象管理（USDF——设备注册、属性标准化、拓扑关系）
> - 接口集成、结果展示、告警闭环流转

## 四、补强方案（修正版）

### 4.1 结构修正（P0）

**新增 §1.2.1 设计前提与分工边界**（插入在 §1.2 方案定位末尾，四约束表之后）：

> **设计前提：双平台分工与算法获取方式**
>
> 本方案与南瑞平台的协作建立在明确的工程分工之上：
>
> | 能力域 | 责任方 | 我方工作 |
> |--------|:---:|------|
> | 功率协同优化算法（下垂控制/有功无功分配优化） | 南瑞算法库 | 通过标准 API 调用 → 结果三维展示 + 安全防误校验 + 指令生命周期管理 |
> | 构网型评价模型（惯量/调频/电压穿越评分） | 南瑞算法库 | 通过标准 API 获取评价结果 → 三维可视化呈现 + 报告自动生成 |
> | 储能诊断算法（内短路/热失控/一致性） | 南瑞算法库 | 通过标准 API 获取诊断结论 → 三维标注 + 告警联动 |
> | CV 视觉 AI（烟火识别/通道隐患/行为识别） | **我方** | 自研 PyTorch→ONNX 边缘推理管线，7×24 实时检测 |
> | 三维空间对象管理（USDF） | **我方** | 核心 IP——设备注册、空间参照、属性标准化、统一 API |
> | 三维可视化渲染与交互 | **我方** | Cesium + Three.js Web 原生引擎，2D/3D 双向联动 |
>
> 这一分工使双方各聚焦自身最强项——南瑞聚焦电力业务算法深度，我方聚焦空间智能与三维交互。两套平台通过 §3.6 定义的页面双向集成 + 算法 API 调用实现无缝协同。

### 4.2 功能设计纵深补强（P0）

选取我方 3 个核心设计域做纵深展开：

**P0-1：AI CV 推理管线设计（补入 §4.4 开头或 §4.4.2 之后）**

新增子节，覆盖：
- 模型选型与部署：YOLO 系列（烟火/隐患/行为） → ONNX 导出 → 场站 II 区 GPU 热加载
- 推理流水线：视频帧采样（10fps）→ 预处理（640×640 resize + 归一化）→ ONNX Runtime 推理 → 后处理（NMS + 置信度过滤 0.5 → 多帧确认 3/5 帧命中才告警）→ 告警写入 Redis pub/sub
- 模型管理：版本注册（USDF 对象模型）、AB 测试框架、模型回滚机制
- 关键参数表：输入分辨率、置信度阈值、NMS IoU 阈值、多帧窗口大小、单帧推理延迟预算（<50ms）

**P0-2：算法接口契约设计（新增 §3.6.3 或独立小节）**

定义南瑞算法库的标准调用契约：
- 接口模式：同步 gRPC（实时控制类，如调频指令计算）/ 异步 Kafka（批处理类，如评价报告生成）
- 统一请求体：`{algorithm_id, device_id, time_range, params{...}}`
- 统一响应体：`{algorithm_id, device_id, timestamp, result{...}, confidence, status}`
- 超时与降级：同步调用 ≤200ms 超时 → 降级为本地兜底策略；异步调用失败 → 自动重试 3 次 → 告警通知
- 调用频率限制：按算法类别分级（暂态类 ≤100ms 间隔 / 稳态类 ≤1s / 评价类按需）

**P0-3：控制指令生命周期状态机（补入 §4.2.1 末尾）**

```
UI 发起指令 → 安全防误三道校验 →
  ├─ 不通过 → 阻断 + 告警通知操作人
  └─ 通过 → 调用南瑞算法计算 →
       ├─ 超时(>200ms) → 降级本地兜底
       └─ 正常返回 → 指令暂存(待人工确认) →
            ├─ 人工确认 → 下发至设备(GOOSE/IEC104) → 执行结果回读 → 状态更新
            └─ 人工驳回 → 指令作废 + 审计记录
```

### 4.3 叙事补位（P1）

- §4.2 开头增加一句：「本章描述的协同优化功能由南瑞算法库提供核心计算，我方三维平台负责策略投退操作界面、指令安全校验、执行结果可视化展示和异常告警联动。」
- §4.3 开头增加一句：「构网型评价的评分模型由南瑞算法库提供，我方负责评价结果的多维度可视化对比、趋势分析和自动报告生成。」
- §4.4 中区分「我方 AI 视觉模型」和「南瑞诊断算法调用」，避免读者混淆。
