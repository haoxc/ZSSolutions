# 平台建设思想：统一空间数据底座 (USDF)

> 原称「一元化空间对象管理」。经方案书调研后正式定名。
> 「底座」定位：它是地基，不是整栋楼——管空间参照系 + 对象注册 + 属性标准化 + 统一 API，不管业务逻辑。
> 远期愿景：从单场站底座演进为多场站统一空间数据操作系统——同一套 API、同一套对象模型、同一套治理体系，支撑集团级数字孪生战略。
> 本文为 JBFG 方案书第三章（总体技术方案）的顶层设计输入。

---

## 一、设计动机

风机三维模型、光伏厂区、储能电池舱、升压站、输电线路——五种空间对象各自为政管理：

```
现状问题:
  风机模型 ≠ 光伏模型 ≠ 储能模型 → 空间坐标系不统一
  巡检数据 / 告警数据 / 运行数据散落 → 跨域查询困难
  新增设备 / 退役设备 → 多处手工同步
  → 后期维护噩梦
```

解法：所有空间对象统一注册到 USDF 底座，标准化属性/关系/状态，业务系统通过统一 API 消费。

---

## 二、方案定位：四个架构约束

> 方案定位：**USDF 统一空间数据底座——新能源场站的空间对象操作系统**。一台设备注册一次，所有业务按需消费。毫秒级实时驱动，AI 模型即插即用。

四个属性不是并列的修饰词，而是架构设计的硬约束条件：

```
可靠                    扩展                  AI友好               实施效能
  │                      │                     │                    │
GOOSE 暂态 ≤100ms     多场站天然支持          USDF 标准 API        边云协同
安全 I/II/III 区        station_id 预留        gRPC/Kafka           断网自持 ≥2h
三通道分流              逻辑多租户              数据就绪              国产化 P0 直选
```

| 属性        | 架构含义                             | 检验标准                   |
| --------- | -------------------------------- | ---------------------- |
| **可靠**    | 暂态控制独立通道，安全分区物理隔离                | GOOSE ≤100ms，I 区防误绝不妥协 |
| **扩展**    | USDF 天然多场站，数据模型一期预留 `station_id` | 二期不加字段、不改 API          |
| **AI 友好** | 北向 API 输出标准化对象 + 时序数据，模型消费即用     | 算法团队不碰采集层              |
| **实施效能**  | 边云协同 + 断网自持 + 国产化 P0 直选          | 一期部署不依赖二期基础设施          |

## 三、USDF 核心定义

### 3.1 定位

```
业务系统层    ← 告警、工单、巡检、报表、AI 诊断（管业务逻辑）
    ↕ 统一 API
USDF 底座层   ← 对象注册、空间参照、属性标准化（管空间对象元数据）
    ↕ 数据管线
数据采集层    ← IEC104/Modbus/GOOSE/GB28181（管数据接入）
```

### 3.2 职责边界

| USDF 负责                             | USDF 不负责   |
| ----------------------------------- | ---------- |
| 所有设备/设施的全局唯一 ID 注册                  | 告警规则引擎     |
| 统一坐标系统 + 层级关系（场站→区域→设备→部件）          | 工单流转逻辑     |
| 每个对象的标准属性集（名称/类型/位置/状态/关联设备/三维模型引用） | AI 推理与模型训练 |
| 向上层业务系统提供统一查询 API                   | 巡检路线规划     |
| 对象生命周期管理（创建/更新/退役）                  | 报表生成逻辑     |
| 空间拓扑关系（包含/邻接/连接）                    | 业务审批流程     |

### 3.3 对象粒度

运维关注的最小单元，非物理最小元件：

| 层级 | 示例 | 粒度 |
|------|------|------|
| 场站 | 冀北风光储基地 | 1 个 |
| 区域 | 风电区、光伏区、储能区、升压站区、输电线路区 | ~10 个 |
| 设备 | 单台风机、单台逆变器、单台储能 PCS | ~500 个 |
| 部件 | 风机叶片、逆变器功率模块 | 按需，非全量 |

**原则**：设备级全覆盖，部件级按需建模。避免百万级对象拖垮底座。

## 四、顶层规划维度

### 4.1 数据分层与流向

```
场站设备
  ├─ 高频采集(1kHz/6.4kHz) ──→ Kafka(分区≥32) ──→ 时序库(ClickHouse)
  ├─ 稳态采集(1s)         ──→ Kafka ──→ 时序库 + 业务库(Postgres)
  └─ 视频流(RTSP/GB28181)  ──→ 流媒体服务器 ──→ 视频存储(NAS)
                                      │
                              ┌───────┴────────┐
实时查询(≤1s)              ↓                  ↓ 历史查询(≤5s)
  Redis 缓存层              展示层（前端API）     物化视图 + 预聚合
```

各层数据契约：

| 层 | 职责 | SLA |
|----|------|-----|
| 采集层 | 协议适配、数据归一化 | 采集间隔 = 数据源频率 |
| 消息总线 | 多源汇聚、分区路由 | 消息延迟 ≤50ms |
| 时序库 | 测点历史、聚合查询 | 写入 100万条/s；查询 ≤5s |
| 业务库 | 主数据、告警、工单 | 事务一致性 |
| 缓存层 | 实时快照 | 命中 ≤1ms |
| 展示层 | API 聚合、权限过滤 | P95 ≤200ms |

**不可越层规则**：展示层不直连时序库 → 采集层不直写业务库 → USDF 不存时序数据。

### 4.2 实时/非实时三通道分流

| 通道      | 协议            | 延迟     | 用途           | 降级策略      |
| ------- | ------------- | ------ | ------------ | --------- |
| P1 暂态控制 | GOOSE         | ≤100ms | 风光储/SVG 紧急控制 | 中断 → 本地兜底 |
| P2 稳态监控 | IEC104/Modbus | ≤1s    | 百万测点实时数据     | 拥塞 → 降采样  |
| P3 历史分析 | RESTful       | ≤5s    | 聚合查询/报表/AI   | 超时 → 近似值  |

### 4.3 边云协同

```
中心云: AI模型训练 / 全局分析 / 跨场站对比 / 历史归档 / 集团驾驶舱
   ↕ 广域网（4G/5G/专线）
场站边缘: 暂态控制(GOOSE ≤100ms) / 本地缓存(Redis) / 断网自主告警
```

断网自持：≥2 小时本地缓存，恢复后自动补齐。

### 4.4 安全分区（电监会 5 号令，等保二级）

```
安全 I 区（控制区）: 暂态控制 + 防误保护
  ↓ 正向隔离装置（物理单向）
安全 II 区（非控生产区）: SCADA采集 + 实时监控 + AGC/AVC + 透明场站 + AI推理
  ↓ 防火墙
安全 III 区（管理信息区）: 运维门户 + 报表 + 三维Web + 大屏
```

### 4.5 多场站扩展

| 阶段  | 策略                                     |
| --- | -------------------------------------- |
| 一期  | 单实例，数据模型预留 `station_id`                |
| 二期  | 逻辑多租户，按 `station_id` 隔离；USDF 天然支持多场站层级 |
| 不变  | 对象模型 / API 契约 / 安全分区架构                 |
| 新增  | 跨场站聚合查询 / 集团驾驶舱 / 场站间协同调度              |

### 4.6 统一 API 与集成标准

南向：IEC104 / Modbus / GOOSE / RTSP-GB28181 / MQTT / RESTful → 统一采集适配器 → Kafka

北向：`/api/v1/objects/{id}` / `/api/v1/objects/{id}/telemetry` / `/api/v1/objects/{id}/alarms` / `/api/v1/objects/{id}/3dmodel`

内部：gRPC（同步） / Kafka（异步） / Nacos（配置）

### 4.7 技术栈国产化

| 层次    | 推荐                  | 国产备选               | 优先级 |
| ----- | ------------------- | ------------------ | :-: |
| 时序库   | ClickHouse          | TDengine/DolphinDB | P0  |
| 业务库   | Postgres+PostGIS    | 达梦DM8/人大金仓         | P0  |
| 消息队列  | Kafka               | RocketMQ/Pulsar    | P0  |
| GIS   | GeoServer           | 超图/中地              | P0  |
| OS    | Linux               | 麒麟/统信              | P1  |
| API网关 | Kong/APISIX         | APISIX（已国产）        | P1  |
| 三维引擎  | Cesium+Three.js（开源） | —                  |  ✅  |
| 流媒体   | ZLMediaKit（开源）      | —                  |  ✅  |
| AI推理  | PyTorch/ONNX        | 昇思/PaddlePaddle    | P2  |

策略：P0 明确推荐国产方案；P1 标注兼容性；P2 可选项。

### 4.8 AI 引擎架构

AI 引擎采用「边缘推理 + 云端训练 + 人机协同闭环」三层架构：

```
边缘端（场站侧）          云端（中心侧）           业务端
  实时 CV 推理             模型训练/微调            人工复核
  告警生成                 模型版本管理             标注反馈
  边缘模型热更新            数据集管理              闭环追踪
      ↕                       ↕                     ↕
      └───────────────────────┴─────────────────────┘
                    USDF 标准 API
                   gRPC（同步）/ Kafka（异步）
```

| 层 | 部署位置 | 职责 | 关键指标 |
|----|---------|------|---------|
| **边缘推理** | 场站 II 区（GPU 服务器） | 烟火识别/通道隐患/设备异常实时检测，结果写入 Redis 缓存 | 单次推理 <500ms |
| **云端训练** | 中心云 III 区（GPU 集群） | 模型微调/数据标注/版本管理/AB 测试 | 模型版本可回溯，新模型下发 ≤30min |
| **人机协同** | III 区（运维门户） | 告警确认/误报标注/模型效果评估 → 反馈至训练集 | 检出率 ≥80%，误检率 30%→15% |

**AI 数据就绪策略**：算法团队不碰采集层——所有训练和推理数据通过 USDF 北向 API 获取（`/api/v1/objects/{id}/telemetry`、`/api/v1/objects/{id}/alarms`）。模型即插即用，新模型接入无需修改数据管线。

**模型范围**：一期覆盖 CV 巡检类模型（F4 智能运维），架构为 NLP/大模型预留接入能力。

### 4.9 数据治理框架

USDF 的属性标准化是数据治理的起点，向上延伸为四层治理体系：

```
┌─────────────────────────────────────────────┐
│  ④ 数据质量                                   │
│  完整性校验 / 时效性监控 / 异常值检测 / 质量报告      │
├─────────────────────────────────────────────┤
│  ③ 数据血缘                                   │
│  测点→Kafka Topic→时序表→API 字段，全链路可追溯     │
├─────────────────────────────────────────────┤
│  ② 元数据目录                                 │
│  场站→区域→设备→部件→测点，五级数据目录 + 搜索        │
├─────────────────────────────────────────────┤
│  ① 主数据管理（USDF 已有）                       │
│  全局唯一 ID + 标准属性集 + 生命周期 + 拓扑关系        │
└─────────────────────────────────────────────┘
```

| 层 | 职责 | 关键规则 |
|----|------|---------|
| **主数据** | 设备/设施全局唯一 ID，标准属性集，生命周期管理 | USDF 3.2 职责边界——单点注册，全局消费 |
| **元数据** | 测点目录、数据字典、API 字段映射、搜索 | 新增测点自动注册到目录，避免「暗数据」 |
| **血缘** | 采集→Kafka→存储→API，全链路字段级溯源 | 支持回溯任意 API 字段的上游数据源 |
| **质量** | 完整性/时效性/唯一性/一致性 四维校验 | 质量 SLA 按通道分级：P1(暂态) > P2(稳态) > P3(历史) |

数据治理与「不可越层规则」互补：不可越层管数据流向的正确性，数据治理管数据自身的可信度。
