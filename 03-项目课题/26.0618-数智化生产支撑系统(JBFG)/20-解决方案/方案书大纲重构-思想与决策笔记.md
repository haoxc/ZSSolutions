# 方案书大纲重构：思想与决策笔记

> 本文记录从 v1 大纲（技术模板型）到 v2 大纲（思想驱动型）的重构逻辑。不是替代品对照，而是**为什么改、改了什么、每条变化的理由**。

---

## 一、核心思想：空间对象操作系统

**一句话定位：** USDF 统一空间数据底座——新能源场站的空间对象操作系统。一台设备注册一次，所有业务按需消费。毫秒级实时驱动，AI 模型即插即用。

这是整个方案书的**元叙事**。它不是一句口号，而是一个精确的类比：

| OS 概念 | USDF 对应 | 含义 |
|---------|----------|------|
| 文件系统 | 空间对象注册 | 每个设备/设施获得全局唯一 ID + 空间坐标 |
| 设备驱动 | 统一采集适配器 | IEC104/Modbus/GOOSE → 统一数据格式 |
| 系统调用 (syscall) | 北向 API | `/api/v1/objects/{id}/*` 标准接口 |
| 进程调度 | 三通道分流 | P1 暂态/P2 稳态/P3 历史，按优先级调度资源 |
| 用户态/内核态 | 业务层/USDF 底座层 | 业务系统不碰数据管线，USDF 不管业务逻辑 |

类比的力量：客户不需要理解「统一空间数据底座」的学术含义，但他们理解「操作系统」——一个让应用程序不用关心底层硬件的平台。

---

## 二、三条底层原理

### 原理 1：对象优先，而非功能优先

传统 SCADA 系统的组织逻辑是**功能树**——按「采集→存储→展示→告警」的管线组织。USDF 的组织逻辑是**对象树**——按「场站→区域→设备→部件」的空间层级组织。

> **推论：** 新增一个业务功能（如「储能电池热失控预警」），不需要重建数据管线。该电池舱早已作为空间对象注册在 USDF 中，新功能只需订阅该对象的遥测数据即可。

### 原理 2：三通道分流——尊重电力物理规律

电力系统的数据天然分三个物理层级，不可混为一谈：

```
P1 暂态控制 (GOOSE ≤100ms)  ← 物理规律：保护动作必须在 100ms 内完成
P2 稳态监控 (IEC104 ≤1s)     ← 物理规律：SCADA polling 周期 1-5s
P3 历史分析 (RESTful ≤5s)    ← 业务规律：报表和 AI 训练不需要实时
```

**这不是「性能优化」，而是「物理约束」。** 试图用同一套数据管道处理三种数据，要么暂态控制不可靠（安全风险），要么稳态监控延迟高（可用性差），要么两者兼顾成本爆炸。

### 原理 3：空间是统一维度

风机、光伏板、储能舱、升压站、输电线路——五种对象看似各异，但共享一个属性：**它们在空间中占据一个位置，与周围对象存在空间关系。** 空间坐标是唯一能将异构对象统一索引的维度。

> **推论：** 跨域查询（「离这台风机最近的储能舱是哪个？」「这条输电线路上有多少块光伏板？」）在功能树系统中需要定制 JOIN 逻辑，在 USDF 中就是一次空间查询。

---

## 三、方法论：从指标反推架构

现有大纲已经实践了这个方法论——从 34 项非功能指标出发，反推出：

```
34 项非功能指标
  ├─ 端到端 ≤1s → 必须三通道分流 (原理 2)
  ├─ GOOSE ≤100ms → 必须安全 I 区独立 (原理 2)
  ├─ station_id 预留 → 必须对象优先 (原理 1)
  ├─ 百万测点 → 必须 ClickHouse + Kafka ≥32 分区
  ├─ AI 检出率 ≥80% → 必须 USDF 标准 API (原理 1)
  └─ 低代码组态 → 平台能力支柱 (Q11)
```

这个方法论应该在方案书中**显式化**——在「建设思想」章节中，展示「每一个架构决策都对应一个可检验的非功能指标」，让客户看到方案的**技术诚意**而非**技术堆砌**。

## 四、与 v1 大纲的对比分析

### 4.1 结构差异一览

| 维度 | v1 大纲 | v2 大纲 | 变化理由 |
|------|---------|---------|---------|
| **卷首语** | 无 | ★ 新增 | 方案书的第一读者可能是领导而非技术负责人——需要 15 秒可读完的「一句话定位 + 差异化 + 量化价值」 |
| **需求理解** | 独立第二章 (SCQA) | 并入 1.2 | SCQA 是项目概述的一部分。客户先知道「我们理解你」再进入「我们怎么做」 |
| **平台建设思想** | 散落在 1.2 (方案定位) + 3.1 (设计理念) | **独立第二章** ★ | 方案书的思想锚点。客户需要能复述「他们的核心思想是什么」——不仅是 USDF 这个词，更是背后的三条原理 |
| **方法论显式化** | 隐含在架构决策中 | 2.6「从指标反推架构」 | 展示技术诚意——每个架构决策都可追溯到一个可检验的指标 |
| **平台能力 (Q11)** | 无 | 2.7「平台能力四个支柱」 | 客户需求澄清中新增的平台级要求，需在方案书中体现 |
| **总体技术架构** | 第三章 (7 节，含 USDF 理念) | 第三章 (9 节，不含思想) | 拆出思想后，架构章聚焦于「怎么做」——技术选型、部署拓扑、数据流向 |
| **业务子系统** | 第四章 (5 节) | 第四章 (5 节) | 保持现有结构 |
| **项目实施** | 第六章 | 第六章 | 保持 |
| **风险管控** | 第七章 (18 项) | 第七章 (18 项) | 保持 |
| **报价与服务** | 第八章 | 第八章 | 保持 |

### 4.2 叙事逻辑对比

```
v1 叙事线：
  项目概述 → 需求分析 → 总体技术方案 → 业务子系统 → 非功能 → 实施 → 风险 → 报价
  (标准「技术方案」模板——平铺直叙，思想隐含在架构中)

v2 叙事线：
  项目概述(含需求理解) → ★平台建设思想(先讲 WHY) → 总体技术架构(再讲 HOW) → ...
  (思想驱动型——先建立认知框架，再用框架解释架构决策)
```

### 4.3 关键变化深度解读

**变化 1：卷首语——「15 秒电梯演讲」**

v1 开篇直接进入 1.1 项目背景。但方案书的实际阅读路径往往是：领导翻第一页 → 找「他们到底要做什么」 → 找「和别家有什么不同」。卷首语就是为这个场景设计的。

**变化 2：平台建设思想独立成章**

这是最核心的结构变化。v1 中：
- 1.2 方案定位写了「空间对象操作系统」的定位
- 3.1 设计理念写了「USDF 底座 + 五业务域」
- USDF 文档写了 9 个顶层规划维度

但三处的关联是**隐含的**——读者需要自己拼凑出「哦，原来三通道分流是因为电力系统有三个物理时间尺度」。v2 第二章将这些关联**显式化**——三条原理、四个约束、一个方法论，构成完整的论证链。

**变化 3：需求理解并入第一章**

v1 第二章 (SCQA) 实际上属于「项目概述」范畴——在讲清楚方案之前，先让客户看到「你理解我的问题」。独立成章反而割裂了「理解→方案」的连贯性。

**变化 4：新增 Q11 平台能力**

这是一个务实补充——客户回填 Q11 后，方案书需要体现这四个平台级能力。放在第二章（思想层）而非第三章（架构层），因为它们是**横向能力**（跨越所有业务子系统），而非单一技术选型。
