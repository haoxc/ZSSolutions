### 2补.3 专项澄清

**F4.3.13 出处「5.2.5.1.2 / 5.2.4.3.8」合理性评估**：

doc 中 5.2.5.1.2 位于「仿真验证管理」章节下，但内容为「光伏场区自动化运维——通过无人机、挂轨机器人搭载可见光和红外成像设备…」，与 5.2.4.3.8「光伏无人化运维」描述同一功能域。这是 doc 自身的编号特殊性：5.2.5.1 仿真验证管理总控把无人机运维作为「仿真验证管理」的输入数据来源之一，故将无人机功能编入 5.2.5.1.x 下。两处描述互补而非重复——5.2.5.1.2 强调「无人机数据接入仿真验证」，5.2.4.3.8 强调「无人机执行运维任务」。

→ **结论**：出处正确，无需修改。F4.3.13 同时引用两个出处合理。

**F4.6 置于 F4（智能运维）下的合理性评估**：

F4.6 运维任务管理（例行/特殊/专项/自定义运维 + 执行/展示/告警确认/审核/报告）是 F4.1–F4.5 各运维场景的**跨域共享层**——它不隶属于任何一个特定场景（变电站/升压站/光伏/输电/储能），而是为所有场景提供统一的任务生命周期管理。

→ **结论**：放在 F4 下作为 F4.6 合理。若作为独立 F7 反而不利于体现「运维任务管理服务于所有智能运维场景」的层级关系。当前编排如下：
```
F4 智能运维
├── F4.1 变电站机房  (场景)
├── F4.2 升压站      (场景)
├── F4.3 光伏        (场景)
├── F4.4 输电线路    (场景)
├── F4.5 储能区      (场景)
└── F4.6 运维任务管理 (共享层——跨所有场景)
```

两个文档在创建阶段存在以下系统性偏差：

1. **参照系错位**：以「方案书的技术选型」（UE5 / 50 CCU / 2 GPU）为分析对象，而非以「doc 的原始需求」为基准。导致非功能分析变成「方案书提案值的可行性评估」而非「客户需求的承接验证」。

2. **可得性启发**：UE5 Pixel Streaming 的 benchmark 数据在公开渠道充分（Epic 官方文档、Streampixel、Tencent Cloud）→ 围绕它做了最详细的分析。而 doc 中更难但对客户更关键的指标（MTBF、测频精度、AI 检出率、接入遥测量）因缺乏现成 benchmark 被跳过。

3. **章节扫描不完整**：功能分解清单覆盖了 §5.2.1–§5.2.4，但 §5.2.5 仿真验证管理（16 项）和 §5.2.4.6–§5.2.4.8 运维任务管理在初次提取时被遗漏——可能因为 §5.2.5 是独立于「生产支撑系统」主线的仿真子系统。
