## M1. 并发用户数 — 50 CCU

### 提案值

> 50 并发用户，2×NVIDIA RTX 6000 Ada 节点。

### Benchmark 校验

**Epic Games 官方文档（UE 5.1 Pixel Streaming Reference）**：
- H.264 NVENC 编码延迟 ~4.6ms/帧（1080p），但 GPU 仅编码**一个会话**——多实例时编码器排队
- 官方未承诺单 GPU 并发数，建议「profiling 决定」

**产业实践（2024–2025）**：

| 来源 | GPU | 单 GPU CCU（1080p@30fps 复杂场景） |
|------|-----|:---:|
| Stephen Phillips (Epic Games DevRel, 2025.07) | — | 「大多数场景 1 GPU = 1 实例」 |
| UIPower (2025) | RTX 3090 | 2–4 |
| Streampixel (2025) | RTX 6000 Ada | 优于消费卡，但仍受 VRAM/编码器约束 |
| Tencent Cloud (2025) | T4/A10 | 1–2（Lumen+Nanite 场景） |
| Epic Developer Forums (2025.09) | 多 GPU 方案 | 单 GPU 上限通常 4，超大规模需编排层 |

### 可行性判定

**❌ 不可行**。2×GPU 支撑 50 CCU 意味着每 GPU 25 实例，是产业上限（4 实例）的 6×+。

GPU 数量的量级估算（保守 3 CCU/GPU，乐观 4 CCU/GPU）：

| 方案 | 所需 GPU |
|------|:---:|
| 保守（3 CCU/GPU） | **17** |
| 乐观（4 CCU/GPU） | **13** |
| 激进 + 低画质预设（6 CCU/GPU） | **9** |

### 风险

- 50 CCU 同时渲染独立 UE5 场景时，VRAM、NVENC 编码器、CPU 均为瓶颈
- Pixel Streaming 会话为**有状态长连接**，无法无状态水平扩展

### 修订建议

| 选项 | 说明 |
|------|------|
| A. 降低 CCU | 一期 8–12 CCU（2 GPU × 4–6 实例），后续按需扩展 |
| B. 增加 GPU 节点 | ≥4×GPU 节点（含 vGPU 支持），匹配 50 CCU |
| C. 混合渲染 | 复杂场景 UE5 Pixel Streaming + 简单场景 Three.js/Cesium（WebGL），降低 GPU 压力 |
| D. 会话池模式 | 非全部 CCU 同时活跃，采用会话池 + 超时回收，峰值 CCU < 注册用户数 |

> **出处**：[Epic Games UE5 Pixel Streaming Reference](https://dev.epicgames.com/documentation/unreal-engine/unreal-engine-pixel-streaming-reference) · [Epic Forums: Best setup for 50–500 users](https://forums.unrealengine.com/t/best-setup-for-running-multiple-avatars-with-pixel-streaming-50-500-users/2636562) · [Streampixel: How to Plan CCUs](https://www.streampixel.io/post/how-to-decide-ccus-for-your-pixel-streaming-app)

## M2. 三维场景帧率 — 30 FPS @1080p

### 提案值

> 30 FPS @1080p，最低可接受帧率。

### Benchmark 校验

**UE5 渲染管线**：30 FPS @1080p 在单实例场景下完全可行（Lumen + Nanite 开启，中高画质预设），前提是 GPU 满足以下条件：
- VRAM ≥ 12GB（中等场景）/ ≥ 16GB（大场景 + 高精度模型）
- 场景三角面 < 100M（含 LOD）
- 无并发编码竞争

**问题不在单会话，而在多会话并发**：50 个 UE5 实例同时渲染 → NVENC 队列深度 50 → 编码帧延迟从 4.6ms 线性增长到不可接受。

### 可行性判定

**⚠️ 有条件可行**。指标本身（30 FPS @1080p）合理，但与 M1（50 CCU）绑定后不可达。若 M1 修订为合理 CCU（8–12），则 M2 可达。

### 修订建议

- 与 M1 联动修订
- 增加 LOD 策略（远景 15fps / 近景 30fps 分级）
- 场景复杂度预算：每个会话三角面 < 50M，DrawCall < 2000

> **出处**：[NVIDIA NVENC Video Encode Guide](https://developer.nvidia.com/video-technologies/nvenc) · UE5 Performance Profiling Guidelines (Epic Dev Community)

## M3. 页面加载时间 — P95 ≤ 200ms

### 提案值

> P95 ≤ 200ms（后台接口）/ 3s（页面整体），复杂报表和历史回放不纳入。

### Benchmark 校验

**Google SRE（2017, ACM Queue）**：
- 用户感知阈值：<100ms = 即时，<1s = 流畅，>1s = 注意力开始漂移
- P95 ≤ 200ms 处于「即时」感知的上界，是合理的核心接口目标

**Spring Boot + Postgres 基准**：
- 简单 CRUD（含索引查询）：5–30ms
- 复杂联查（多表 JOIN + 空间查询）：50–150ms
- 聚合报表（跨月/跨年）：500ms–5s

**分层策略是正确的**：核心接口 200ms + 报表放行，这是 Google SRE 推荐的「差异化 SLO」实践。

### 可行性判定

**✅ 可行**。
- 200ms 对 Spring Boot + Postgres（含 PostGIS）可达
- 关键前提：连接池配置、索引优化、Redis 缓存热点数据、读写分离
- 3s 页面整体加载含三维场景首次加载——UE5 Pixel Streaming 首次连接通常在 2–5s，需预案

### 注意事项

- PostGIS 空间查询（ST_Intersects、ST_Distance）在大数据量下可能超过 200ms，需物化视图/缓存
- 2000+ 测点实时写入与查询竞争 → 读写分离为主备，查询走只读副本

> **出处**：[The Calculus of Service Availability — Google SRE (ACM Queue, 2017)](https://queue.acm.org/detail.cfm?id=3096459) · Spring Boot Performance Tuning Guide · PostgreSQL Query Performance Guide

## M4. 实时数据刷新延迟 — ≤500ms

### 提案值

> 端到端延迟 ≤500ms：SCADA → 数据网关 → Kafka → 后端处理 → 数据库 → API → WebSocket → 前端渲染。

### Benchmark 校验

**SCADA 层延迟**：

| 协议 | 典型延迟 |
|------|:---:|
| GOOSE（IEC 61850 保护类） | 1–3 ms |
| MMS（IEC 61850 SCADA 遥测） | 10–100 ms |
| ModBus TCP 轮询 | 50–200 ms/轮 |
| IEC 104 轮询 | 100–500 ms/轮 |

**中段延迟**（Kafka + 处理 + DB 写入）：
- Kafka 端到端（produce → consume）：**5–15ms**（同机房，batch size 优化）
- 后端处理 + DB 写入：**10–50ms**（批量写入）
- WebSocket 推送：**5–20ms**（局域网）

**前端渲染延迟**：
- Three.js/Cesium 数据→可视更新：**16–33ms**（1–2 帧 @60fps）
- UE5 Pixel Streaming 接收+解码+渲染：**30–50ms**

**全链路汇总**：

| 链路 | 乐观 | 典型 | 悲观 |
|------|:---:|:---:|:---:|
| SCADA 采集 (IEC 104) | 100ms | 200ms | 500ms |
| Kafka → 处理 → DB | 20ms | 50ms | 150ms |
| API → WebSocket → 前端 | 30ms | 50ms | 100ms |
| **合计（不含 UE5）** | **~150ms** | **~300ms** | **~750ms** |
| + UE5 Pixel Streaming | +50ms | +80ms | +150ms |
| **合计（含 UE5）** | **~200ms** | **~380ms** | **~900ms** |

### 可行性判定

**⚠️ 有条件可行**。典型场景 300ms 落在 500ms 以内，但悲观场景（IEC 104 轮询慢 + 网络抖动 + UE5 解码延迟）会超过 500ms。

### 关键约束

- IEC 104 轮询周期必须 ≤ 500ms（即至少 2Hz 采集频率）
- Kafka 必须与后端同机房部署，避免跨网络延迟
- UE5 Pixel Streaming 增加 50–150ms 不可控延迟（编解码+网络），不适合「实时」场景；若对延迟要求高，UE5 渲染应采用本地客户端而非 Pixel Streaming
- **与 M1 联动**：50 CCU 时 Kafka consumer 竞争加剧，中段延迟恶化

### 修订建议

- 区分「软实时」（≤ 2s，适合 Pixel Streaming 远程渲染）和「硬实时」（≤ 500ms，需要本地渲染）
- 或将 ≤500ms 限定为「后台数据管线延迟」，不含前端渲染

> **出处**：[IEC 61850 Communication Networks and Systems](https://scadaprotocols.com/process-bus-vs-station-bus-iec-61850/) · [TU Delft: Real-Time Implementation of Virtualized IEC 61850 Controllers](https://pure.tudelft.nl/ws/portalfiles/portal/218890510) · Apache Kafka Performance Tuning (Confluent)

## M5. 系统可用性 SLA — 99.99%

### 提案值

> 99.99%（四九），核心业务链路（实时状态、设备画像、告警查询、派单接口）。

### Benchmark 校验

**Google SRE 四九架构铁律**（Treynor, Dahlin, Rau, Beyer — *The Calculus of Service Availability*, ACM Queue 2017）：

| 铁律 | 要求 | JBFG 现状 |
|------|------|-----------|
| 1. 无单点故障（SPOF） | 每层 ≥2 节点，跨≥2 可用区 | 应用层满足，**GPU 渲染层不满足**（2 节点中任一宕机 → 50% CCU 丢失） |
| 2. 关键依赖 +1 九 | 所有关键依赖需达 99.999% | UE5 Pixel Streaming 不是高可用服务——它是有状态单例进程 |
| 3. 检测 < 2 分钟 | 外部分钟级探活 | 可实现 |
| 4. 自动恢复 | 数据库自动故障切换、进程自愈 | 数据库可配 Patroni 自动故障切换。GPU 渲染会话**无法自动恢复**——会话状态丢失 |
| 5. 年故障预算 | 52 分钟/年 | 一次 GPU 驱动崩溃重启（~3 分钟）× 2 GPU × 月度频率 = 72 分钟/年，已超预算 |

**UE5 Pixel Streaming 的 SPOF 问题**：
- 每个 Pixel Streaming 会话绑定到一个 UE5 进程，进程绑定到一个 GPU
- GPU 故障 → 该 GPU 上所有会话终止 → 用户必须刷新重连
- 这是**有状态单例架构**，与四九 SLA 要求的无状态/自动故障切换根本冲突

**年故障预算实际消耗估算**：

| 故障源 | 频率 | 单次时长 | 年累计 |
|--------|:---:|:---:|:---:|
| UE5 进程崩溃 | 每月 1 次/GPU | 2 min | 48 min |
| GPU 驱动崩溃 | 每季 1 次 | 5 min | 20 min |
| Postgres 故障切换 | 每年 2 次 | 30 sec | 1 min |
| Spring 应用重启 | 每月 1 次 | 30 sec | 6 min |
| 网络抖动 | 每月 1 次 | 1 min | 12 min |
| **合计** | | | **~87 min** |
| **四九预算** | | | **~52 min** |

> **结论：仅 UE5 层故障就消耗 68 分钟/年，超过四九 52 分钟预算。**

### 可行性判定

**❌ 不可行**。UE5 Pixel Streaming 的有状态架构与四九 SLA 在架构层面不可调和。

### 修订建议

| 选项 | SLA | 年故障预算 | 适配性 |
|------|:---:|:---:|:---:|
| A. 分层 SLA | 后台服务 99.99% + 渲染层 99.9% | 后台 52min + 渲染 8.8h | ✅ 推荐 |
| B. 整体三九 | 全局 99.9% | 8.8h | 可接受（含 GPU 故障容忍） |
| C. 保留四九 + 去 UE5 | 后台 99.99%，前端用 WebGL（Three.js/Cesium） | 52min | 架构全面改动 |

**推荐 A**：后台业务链路（Spring + Postgres + Kafka）四九，三维渲染层三九。这是 Google SRE 推荐的「差异化 SLO」模式——不同链路不同目标。

> **出处**：[The Calculus of Service Availability — Google SRE (ACM Queue, 2017)](https://queue.acm.org/detail.cfm?id=3096459) · [LinkedIn School of SRE: Availability](https://linkedin.github.io/school-of-sre/level101/systems_design/availability/) · [UE5 Pixel Streaming Reference (Epic Games)](https://dev.epicgames.com/documentation/unreal-engine/unreal-engine-pixel-streaming-reference)

## M6. 数据存储周期 — 3 年

### 提案值

> 3 年在线存储，离线归档按需延长。

### Benchmark 校验

**数据量级估算**：

| 数据类型 | 粒度 | 日增量 | 3 年累计（含 2×压缩） |
|---------|------|:---:|:---:|
| 时序测点（2000 点，1s） | 50B/点 | ~8.6 GB | ~9.4 TB |
| 告警记录 | ~1000 条/天 | ~1 MB | ~1 GB |
| 视频录像（100 路，事件触发存储） | 按 20% 路数持续录 | ~400 GB | ~438 TB |

**存储配置对照**：

| 存储层 | 配置 | 容量 |
|--------|------|:---:|
| DB 服务器 ×2 | 2×1.92TB NVMe + 4×8TB SATA（RAID10） | 16TB 可用/节点 |
| NAS | 50TB 万兆 | 50TB |

**问题**：视频录像 438TB 远超 NAS 50TB 容量。

### 可行性判定

**⚠️ 时序数据可行，视频不可行**。

| 数据类型 | 判定 |
|---------|:---:|
| 时序测点（9.4 TB） | ✅ 可存入 DB 服务器 SATA 阵列或 NAS |
| 告警记录（1 GB） | ✅ 微不足道 |
| 视频录像（438 TB） | ❌ 4×8TB/节点完全不够 |

### 修订建议

- 视频存储：仅保留 30–90 天循环覆盖，历史视频离线归档到磁带/蓝光/云存储
- NAS 扩容至 ≥100TB（若需 90 天视频）或添加专用视频存储阵列
- 时序数据：3 年在线是合理的（~10TB），推荐 TimescaleDB 自动分区 + 压缩

> **出处**：PostgreSQL Storage Sizing Guide · TimescaleDB Compression Benchmarks · GB/T 28181 录像存储规范

## M7. 视频接入路数 — 100 路

### 提案值

> 100 路 RTSP/GB28181，1×流媒体服务器（16 核 + 64GB + 4×8TB SATA）。

### Benchmark 校验

**产业基准**（GB28181/RTSP 转发场景）：

| 来源 | 路数 | CPU | 内存 | 带宽 |
|------|:---:|-----|:---:|:---:|
| E-COM-NET 国标平台 | 100 | 4 核 | 8 GB | 200 Mbps（720p） |
| 达讯 SVMSLite | 100 | 4 核 | 16 GB | 250 Mbps |
| 宇视 VS-MS8800 | 768 Mb 入口 | 8 核 | 8 GB | 768 Mbps |
| wvp-GB28181-pro (开源) | 50+ | 8 核 | 16 GB | 千兆 |
| 百度开发者指南 | 100 (4K) | 16 核 | 64 GB | 1 Gbps |

**JBFG 提案配置对照**：16 核 + 64GB → 匹配百度 4K 100 路方案，对 1080p 场景**绰绰有余**。

**带宽估算**：

| 分辨率 | 单路码率 | 100 路带宽 |
|--------|:---:|:---:|
| 720p | 2 Mbps | 200 Mbps |
| 1080p | 4 Mbps | 400 Mbps |
| 4K | 8 Mbps | 800 Mbps–1 Gbps |

### 可行性判定

**✅ 可行**。16 核 + 64GB + 万兆网络的配置对 1080p 100 路纯转发有充足余量。

### 注意事项

- 仅适用于**纯转发**（不解码/不转码/不做 AI 分析）
- 若需实时转码（如 RTSP → WebRTC 供浏览器播放），每路增加 1–2 CPU 核心负载 → 需额外 GPU 硬编码或增加 CPU 节点
- 4×8TB SATA → 1080p@4Mbps 100 路持续录像：~3.5 TB/天，存储仅够 ~9 天连续录像。需与 M6 联动——明确视频留存策略

> **出处**：[wvp-GB28181-pro 部署指南](https://github.com/648540858/wvp-GB28181-pro) · 宇视 VS-MS8800 规格书 · 达讯 SVMSLite 硬件要求 · [GB/T 28181-2016 标准](https://openstd.samr.gov.cn/)

## 风险矩阵

| # | 风险 | 等级 | 影响 | 缓解措施 |
|---|------|:---:|------|---------|
| R1 | 50 CCU 需 2 GPU → 实际需 9–17 GPU | 🔴 阻断 | 方案书 GPU 配置严重低估，报价/部署均不可行 | M1 修订为 8–12 CCU 或增加 GPU 至 ≥4 节点 |
| R2 | UE5 + 四九 SLA 架构冲突 | 🔴 阻断 | 合约 SLA 无法兑现，验收风险 | 分层 SLA（后台四九 + 渲染三九） |
| R3 | 端到端延迟悲观场景超 500ms | 🟡 高 | 实时性不达标，客户体验差 | 区分管线延迟 vs 渲染延迟，明确定义 |
| R4 | 视频存储容量 50TB vs 需求 438TB/3 年 | 🟡 高 | 存储爆满导致录像丢失 | 视频仅保留 30–90 天热数据，历史归档 |
| R5 | 30 FPS 在 50 CCU 并发时不可达 | 🟡 高 | 与 M1 联动 | 与 R1 同步缓解 |
| R6 | Pixel Streaming 首次连接 2–5s > 3s 页面目标 | 🟢 中 | 用户感知慢 | 加载态 UI + 渐进式渲染 |
