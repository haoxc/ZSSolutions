# 26.0619-需求分析深度审计-QIE-DEEP

> 本记录用于个人复盘，不作为项目内容、正式证据或交付物。

## 结论

Q 评估指出审计方案的结构性缺陷：以方案书的技术选型（UE5 Pixel Streaming）为参照系，而非以 doc 原始需求为第一性基准——典型的锚定效应。核心整改方向：将两份需求分析产物从「评估方案书提案值」重构为「忠实承接 doc 原文需求」，去除 UE5 思维定势，补全缺失章节（§5.2.5 仿真验证管理 + §5.3 全部量化指标 + §5.4 接口要求）。整改后非功能指标定义不再为特定技术栈辩护或设限，功能分解清单 C-E 完整。

## 专家评估（Phase 3 Q）

### 判定
方案合理，但有一个结构性缺陷：审计框架以「非功能指标定义」的分析结论为参照系（UE5 Pixel Streaming 为前提），而非以 doc 原始需求为第一性基准。

### 心智模型诊断
1. **锚定效应 (Tversky & Kahneman)**：「50 CCU」是锚。doc 原文是「并发用户数 ≥500 用户」，两者鸿沟被「CCU 只算渲染会话」的隐含假设填平。
2. **Cynefin 框架 (Snowden)**：用繁杂域方法（逐项评估定量指标）处理复杂域问题（架构选型+技术路线+客户期望三层嵌套）。
3. **MECE 原则 (McKinsey)**：功能分解是 MECE 的（除 §5.2.5 缺失），但性能指标覆盖不互斥——7 个指标是 doc §5.3 的子集，筛选标准未定义。

### 多维评估表
| 维度 | 判断 | 说明 |
|------|------|------|
| 合理性 | ⚠ | 三维审计框架本身合理，但技术基线选错了参照系——应以 doc 为基准 |
| MECE | ✗ | 非功能分析覆盖不完整；§5.4 接口要求零覆盖 |
| 隐含假设 | ① UE5 是既定技术路线（doc 未指定） ② 50 CCU 从 500 用户合理推导（未经论证） ③ 非功能分析的 7 个指标是「最重要」的（无筛选依据） |
| 认知偏差 | 锚定：把方案书提案值当审计基线。可得性：UE5 benchmark 数据充分→围绕它做最细分析，更难但对客户更重要的指标因缺现成 benchmark 被略过 |
| 可执行性 | ✅ | 审计方案可落地。产出物边界清晰 |

### 定势突破
1. **翻译稿类比**：审计需求分析产物应逐句对照原文——漏译、错译、增译都是问题。当前审计在查「错译」和「漏译」，但「增译」（方案书自行引入的技术约束）需要单独标注。
2. **倒置思维**：最可能的失败模式是审计报告出来后团队说「架构已定」，审计变成墙上的纸。应对：按阻断/高危/中危/建议分级，阻断项给出不可回避的决策闸门。

### 替代方案
1. 增设第一性基准线——审计报告中单独成章「Doc 需求还原」，先原样列出客户需求再标注承接情况
2. 区分「客户需求」和「方案假设」——新增「自增约束清单」
3. 非功能分析补全——用全量对照表，不再「挑拣」指标
4. 风险矩阵三维独立成表——技术/业务/实施各自纵轴
